Capstone Projects

Every chapter so far has ended with a project you could finish in an hour or two. That was deliberate — each one was scoped tightly around that chapter's own skill. A capstone is different on purpose: it's bigger, it takes days instead of hours, and nobody hands you a starter file with the shape already decided. This chapter doesn't teach new Python syntax at all — it teaches the planning skill that actually determines whether a multi-day project ships or stalls out half-finished: how to scope it, how to combine skills from chapters you've already learned, and how to write it up so it's genuinely worth putting in front of an employer.

  • A note on this chapter's shape: there's no exercises/, no practice/, and no challenges.html here — those formats are built for auto-graded, single-sitting problems, and a multi-day capstone build doesn't fit that mold. Sub-topic 1 of 6

1. What makes a portfolio project worth showing

At a glance
  • "It works" is the minimum bar, not the goal — a reviewer assumes your code runs; what they're actually evaluating is the decisions behind it.
  • A project that looks identical to a well-known tutorial signals "I followed instructions," not "I can build something."
  • The signal that actually reads as hireable: real scope decisions, a real (or realistic) use case, some tests, and a README that explains why, not just what.
Why this matters: most self-taught portfolios are full of technically-working projects that all fail the same way — they prove you can type code someone else already designed, not that you can make the calls a working developer makes every day: what to build first, what to leave out, and how to explain a trade-off honestly.

"I followed a tutorial" and "I made real decisions" can produce visually similar code — the difference is invisible in a screenshot and only shows up when you look closer:

  • Scope was chosen, not copied. The feature list matches a real problem the builder cared about, not a fixed checklist from a course.
  • The data or use case is real, or realistically framed. Not three rows of placeholder "foo"/"bar" data — a dataset, input format, or workflow that resembles something an actual user would hit.
  • Something is tested. Not necessarily exhaustive coverage — but the tricky logic (the part most likely to have a bug) has at least a few tests proving it, the way Chapter 27 covered.
  • The README explains reasoning, not just usage. "Here's how to run it" is necessary but not sufficient — "here's why I stored data this way instead of the obvious alternative" is what actually differentiates the project.
What is... a portfolio project? In a hiring context, a portfolio project is any piece of work a candidate points to as evidence of real ability — distinct from a resume bullet point, which is just a claim. Its entire value is that it's checkable: a reviewer can open the repo, read the code, and see for themselves whether the claims on the resume hold up. That's also exactly why a copy-of-a-tutorial project undercuts itself — the moment a reviewer recognizes the tutorial, the "checkable evidence" becomes evidence of the tutorial's quality, not yours.

None of this means a capstone has to be original in some grand sense — almost every project idea has been built before, many times. What has to be original is the specific set of decisions: which features made the cut, which data shape you chose, which trade-off you accepted and can explain. Two people can build "a task API" from the exact same course and end up with genuinely different, individually defensible projects.

Quick recap
  • "It works" is table stakes — what actually gets evaluated is the decisions behind the code.
  • Real scope, a realistic use case, some tests, and a reasoning-first README are what separate a capstone from a copied tutorial.
  • Originality lives in the decisions, not in inventing a never-before-built idea.
Sub-topic 2 of 6

2. Scoping a capstone

At a glance
  • A capstone needs a roadmap before it needs code: a small, genuinely finishable core (the MVP), plus an explicit list of stretch goals you're allowed to skip.
  • Time-boxing — deciding up front how many days or hours a phase gets — is what stops "just one more feature" from eating the whole project.
  • Over-scoping a first capstone is the single most common way an ambitious learner ends up with nothing finished to show at all.
Why this matters: an unfinished, over-scoped project is worth less in a portfolio than a small, finished one — a reviewer can't evaluate decisions in code that doesn't run yet.
What is... an MVP? MVP (Minimum Viable Product) is the smallest version of an idea that's genuinely usable end-to-end — every core feature present, nothing polished, nothing "extra." For a task API, that's create/read/update/delete on one resource with real persistence, not five resources with three half-built. The point of naming it explicitly is that it becomes a real checkpoint: "is the MVP done?" has a yes/no answer, while "am I making good progress?" doesn't.
What is... time-boxing? Time-boxing means assigning a fixed amount of time to a phase of work in advance, and moving on when that time is up — whether or not the phase feels "done." Applied to a capstone: "MVP gets 3 days, stretch goal 1 gets 1 day, README and tests get 1 day" — not "I'll build until it feels finished," which for a perfectionist rarely arrives on schedule.

A roadmap that actually works has three tiers, in this order, and the order matters — the MVP has to be genuinely runnable before any stretch goal is touched:

  • MVP — the smallest end-to-end slice. It should be boring to describe: "users can add, view, update, and delete a task." No auth, no extra endpoints, no nice-to-haves yet.
  • Stretch goals — a ranked list of what you'd add next, if time allows. Ranked matters: if only one stretch goal gets built, it should be the one at the top, not whichever one happened to sound fun that day.
  • Explicitly out of scope — things a capstone in this space could plausibly include but that you're deliberately not attempting (e.g. "no user authentication system, no deployment to a live server"). Writing this list down does real work: it's what turns "I didn't get to it" into "I chose not to," which reads very differently in a README.

The over-scoping trap is specific and predictable: a first capstone idea almost always starts as "a full app with users, auth, a database, a REST API, a frontend, deployment, and tests" — six chapters' worth of skills, attempted simultaneously, with no MVP checkpoint along the way. The fix isn't to want less — it's to sequence the same ambition: build the MVP first, get it fully working and tested, and only then decide whether there's still time left for the rest.

Thought process: an AI assistant is genuinely useful here, but only if you give it your actual constraints (time available, skills you have vs. don't) — asking it to just "help me plan this project" tends to produce an idealized roadmap with no MVP checkpoint at all, the exact failure mode this sub-topic is warning about.

Write the prompt you'd give an AI assistant to help scope your capstone:

Paste your prompt into Claude, ChatGPT, Copilot, or whatever assistant you use, and compare what it proposes against this sub-topic's three-tier structure — does it actually separate MVP from stretch goals, or does it hand back one flat feature list with no sequencing at all?
Show a strong version of this prompt 
I'm scoping a capstone project: [describe your idea in 2-3 sentences]. I have roughly [X days / hours] total to work on it, and I already know [list the skills/chapters you're comfortable with]. Break this into three tiers: (1) a genuinely minimal MVP that's usable end-to-end with no extras -- describe exactly what it can and can't do, (2) a ranked list of 3-5 stretch goals, ordered by which would add the most value if I only have time for one, (3) an explicit "out of scope" list of things this kind of project could plausibly include that I should deliberately skip. Also suggest a rough time-box for the MVP phase alone, given my total time budget.
  • Every capstone roadmap needs three tiers: MVP, ranked stretch goals, explicit out-of-scope.
  • Time-boxing each phase (a fixed number of days, decided up front) is what actually protects the schedule from scope creep.
  • Over-scoping is the default first instinct — the fix is sequencing the same ambition behind an MVP checkpoint, not shrinking the ambition itself.
Go Deeper: what if I finish the MVP faster than expected?

Move straight to stretch goal 1, in ranked order — don't restart planning from scratch. A roadmap finishing ahead of schedule is a good problem to have, but it's still worth resisting the urge to add an unranked, unplanned feature just because there's time; if it's genuinely valuable, it belongs on the stretch-goal list, ranked against the others, not slipped in as an afterthought.

Quick recap
  • A roadmap has three tiers: MVP, ranked stretch goals, explicit out-of-scope.
  • MVP = the smallest genuinely end-to-end version; time-boxing keeps each phase from running long.
  • Over-scoping a first capstone is the most common reason ambitious learners finish nothing.

3. Combining skills from multiple chapters

At a glance
  • A capstone is defined less by a single new technique than by which earlier chapters' skills it composes into one working system.
  • A useful way to scope an idea: name the 3-5 chapters it draws from, and check that each one is actually load-bearing, not just mentioned.
  • If an idea only really uses one chapter's skill, it's a regular project, not a capstone — capstones exist specifically to prove skills work together.
Why this matters: a real job rarely uses one skill in isolation — a backend feature touches a database, an API layer, error handling, and tests all in the same pull request. A capstone is the first time this course asks you to hold that many moving pieces at once on purpose.

A simple way to find a capstone-worthy idea: list a few earlier chapters whose skills genuinely go together, and see what they add up to.

example skill combinations 
Databases (Ch 20-21) + Web APIs (Ch 22) + Testing (Ch 27)
    -> a real backend service with persistence, an HTTP layer, and a
       test suite proving both work together

File/CSV handling (Ch 19) + Data structures (Ch 8-9) + CLI design (Ch 31)
    -> a command-line tool that ingests real data files and reports on
       them, packaged the way a real CLI tool ships

Error handling (Ch 13) + Logging (Ch 31) + Automation/scripting
    -> an "ops toolkit" that behaves safely when things go wrong and
       leaves a trace of what it did

OOP (Ch 14-16) + File persistence (Ch 19) + Testing (Ch 27)
    -> a stateful application (tracker, simulator, small game) that
       saves and reloads its own state reliably

The check that matters isn't "did I mention four chapters in my README" — it's whether each one is actually load-bearing: if you deleted that chapter's skill from the project, would something genuinely break or disappear? A task API that happens to store data in a file it never really needs isn't "combining file handling and web APIs" — it's a web API with an incidental file next to it. A task API whose tests specifically catch a bug in how concurrent requests touch that file is combining them for real.

What is... a "load-bearing" skill? Borrowed from construction: a load-bearing wall can't be removed without the structure failing. Applied to a capstone, a chapter's skill is load-bearing if removing it would break something the project actually depends on — as opposed to a skill that's technically present but doesn't do real work. Naming which chapters are load-bearing in your own project (and being honest when one isn't) is itself part of the planning skill this chapter is teaching.
  • A capstone idea should draw on 3-5 earlier chapters' skills, not one.
  • Each named skill should be load-bearing — genuinely necessary, not just present.
  • This is exactly how the 5 capstones in this course (Sub-topic 5) were each designed: around a specific, named skill combination.
Quick recap
  • Capstones are defined by which chapters' skills they compose, not one standout technique.
  • List the 3-5 chapters an idea draws from, and check each is load-bearing.
  • An idea using only one chapter's skill is a regular project, not a capstone.

4. Writing a capstone README that gets you hired

At a glance
  • A README that only explains how to run a project answers the least interesting question a reviewer has.
  • The README sections that actually move a hiring decision: the problem, the architecture decisions and why, honest limitations, and what you'd change for production.
  • An honest limitations section reads as more senior, not less capable — it's evidence you know what "production-ready" actually requires.
Why this matters: a hiring reviewer skims dozens of repos; "how to run it" is scannable in five seconds from the code itself, but "why did you build it this way" only exists if you write it down — and it's the part that actually differentiates candidates.

Here's the anatomy, roughly in the order a reviewer actually reads it:

  • Problem statement. One or two sentences: what real situation does this solve, for whom? Not "a task manager" — "a task API for a small team that needed status tracking without paying for a project-management SaaS."
  • Architecture & decisions. The choices that weren't obvious, and why you made them — "used SQLite instead of a hosted database because this runs locally and doesn't need concurrent multi-user writes," not a restated list of the tech stack.
  • How to run it. Exact commands, in order, that actually work if copy-pasted — the boring-but-necessary section.
  • What you'd do differently in production. Name the corners you cut on purpose to hit scope — no auth, no rate limiting, in-memory instead of a real queue — and what you'd add if this were shipping for real.
  • Honest limitations. What doesn't work yet, what's untested, what edge cases you know are unhandled. This is the section most learners are tempted to skip, and it's the one that most reads as senior when it's there.
Example

A directory layout that matches this anatomy — the README sits at the root, not buried where a reviewer has to go looking for it:

task-api/
├── README.md              <- problem, architecture, how to run,
│                              production notes, limitations
├── requirements.txt
├── app/
│   ├── __init__.py
│   ├── models.py
│   ├── routes.py
│   └── db.py
├── tests/
│   └── test_routes.py
└── .gitignore

Notice what's not in that anatomy: a wall of badges, a feature list copied from the code's docstrings, or a "future work" section that vaguely gestures at "add more features." Every section above answers a specific question a reviewer actually has — "what problem," "why built this way," "does it run," "do you know what's missing," in that order.

Thought process: asking an AI assistant to "write my README" tends to produce generic boilerplate padded around a feature list — the useful move is asking it to interview you, or to critique a draft you already wrote against this specific five-part anatomy, so the reasoning stays yours and the AI's job is catching gaps, not inventing content.

Write the prompt you'd give an AI assistant to get feedback on your capstone README:

Paste your prompt into Claude, ChatGPT, Copilot, or whatever assistant you use, along with your actual draft README, and compare its critique against your own gut sense of what's weak — a good critique should point at specific missing reasoning, not just suggest generic additions like badges or a table of contents.
Show a strong version of this prompt 
Here's my capstone README draft: [paste it]. Review it against this five-part anatomy: (1) a one-to-two-sentence problem statement naming a real situation and audience, (2) an architecture/decisions section explaining WHY non-obvious choices were made, not just listing the tech stack, (3) exact, copy-pasteable run instructions, (4) an honest "what I'd do differently in production" section naming corners cut on purpose, (5) an honest limitations section naming what's untested or unhandled. Tell me which of these five sections is missing or too vague, and for each weak section, ask me a specific question that would help me write a stronger version -- don't just write the content for me.
  • A README's most valuable sections are the ones code alone can't answer: why, not just what.
  • Anatomy: problem statement, architecture decisions, how to run, production trade-offs, honest limitations.
  • An honest limitations section is a signal of seniority, not a confession of failure.
Go Deeper: how long should a capstone README actually be?

Long enough to cover all five sections with real content, short enough that a reviewer reads the whole thing in under two minutes — usually somewhere between half a page and a full page of rendered Markdown. A ten-page README is nearly as bad a signal as a two-line one: it usually means padding, not depth. If a section has nothing substantive to say (a truly trivial project might have no real "production trade-offs"), it's fine to say so briefly rather than stretching it.

Quick recap
  • "How to run it" is necessary but not the section that differentiates a capstone.
  • Cover problem statement, architecture decisions, how to run, production trade-offs, and honest limitations.
  • Admitting what's unfinished or untested reads as senior judgment, not a weakness to hide.

At a glance
  • This course ships 5 full capstone roadmaps, each combining a different set of earlier chapters' skills.

  • You're not expected to build all 5 — Sub-topic 6 covers how to choose.

Each capstone below has its own roadmap doc with the full MVP / stretch-goal / structure breakdown — this is just the pitch for each, and the chapters it leans on hardest.

This is the closest of the five to a standard "backend engineer" portfolio piece, and the one most directly transferable to a junior backend or full-stack role.

6. How to pick which capstone(s) to build

At a glance
  • Build at least one capstone all the way to a finished MVP; two, with genuinely different skill combinations, makes a stronger portfolio than one.
  • Choose based on the job or domain you're actually targeting, not which one sounds most impressive in the abstract.
  • Partially completing one is fine — as long as your README is honest about what's done versus planned, per Sub-topic 4.
Why this matters: five finished capstones would obviously make the strongest portfolio, but "obviously strongest, unrealistic to finish" is worse advice than "genuinely finishable" — this course would rather you ship one honestly-scoped capstone than start five and finish none.

A simple way to choose: match the capstone's skill combination to the kind of role you're targeting.

  • Targeting backend/full-stack roles? The Full-Stack Task API is the most directly relevant — it's the closest match to what an actual junior backend ticket looks like.
  • Targeting data/analytics-adjacent roles? The Data Insights CLI demonstrates comfort with real datasets and turning them into something usable, without needing a full data-science stack this course hasn't covered.
  • Targeting DevOps/SRE/infrastructure-adjacent roles? The Ops Toolkit is the most on-theme — it's literally the category of script that role writes daily.
  • Want to show sustained, careful modeling of a real domain? The Personal Finance Tracker rewards getting a data model right over time, which reads well for roles that care about correctness and data integrity.
  • Want to prove you can work inside someone else's code, not just your own? The Open-Source Contribution Simulation is unique among the five for testing exactly that — genuinely valuable no matter what role you're targeting, since almost every real job is mostly this.

If you can't decide, build two with deliberately different skill combinations — say, the Task API (backend-heavy) and the OSS Simulation (code-reading-heavy) — rather than two that mostly overlap. Two capstones that each prove a different kind of competence tell a reviewer more than two that both prove the same one twice.

What is... a "scoped-down" capstone? A scoped-down capstone is a capstone whose README honestly narrows its own MVP claim to match what actually got built — e.g. "this README originally planned full CRUD plus authentication; what's actually finished is CRUD without auth, documented below as a stretch goal I ran out of time for." This is not a failure state. A capstone that's honestly scoped down and fully working beats one that claims everything from Sub-topic 5's pitch but has broken or missing pieces a reviewer discovers by actually running it.

Thought process: the useful AI-assisted move at this stage isn't "which capstone should I pick" in the abstract — it's giving the assistant your actual constraints (time available, target role, current comfort with each chapter) so the suggestion is grounded, then treating its answer as one input to your own decision, not the decision itself.

  • Pick based on the role/domain you're targeting, matched against each capstone's skill combination.
  • A scoped-down, honestly-documented partial capstone beats an over-claimed one that doesn't hold up when run.
In the Real World
  • Every real engineering project starts as a scoping exercise before a single line of code is written — sprint planning, RFCs, and design docs are all the professional version of the MVP/stretch-goal/out-of-scope roadmap this chapter teaches.
  • Hiring managers and technical interviewers routinely ask "walk me through a project you built and why you made the decisions you did" — a capstone with a strong README is quite literally rehearsed material for that exact question.
  • "What would you do differently in production" is one of the most common interview questions asked about a candidate's own portfolio project — because it tests whether you actually understand the gap between "works on my machine" and "ready to ship," not just whether the code runs.

Points to Remember

  • A portfolio project is judged on the decisions behind it, not just whether it runs — real scope, a realistic use case, some tests, and reasoning in the README are what separate it from a copied tutorial.
  • Scope every capstone in three tiers: a genuinely minimal MVP, a ranked list of stretch goals, and an explicit out-of-scope list — time-box each phase to protect against over-scoping, the most common reason ambitious first capstones go unfinished.
  • A capstone is defined by which earlier chapters' skills it composes together, not one standout technique — each named skill should be load-bearing, genuinely necessary to the project, not just present.
  • A hireable README covers five things a reviewer actually wants to know: the problem, the architecture decisions and why, how to run it, what you'd change in production, and an honest limitations section.
  • This course ships 5 capstones — Full-Stack Task API, Data Insights CLI, Ops Toolkit, Personal Finance Tracker, and Open-Source Contribution Simulation — each with its own roadmap doc and a different skill combination.
  • Build at least one capstone to a finished MVP, ideally two with different skill combinations, chosen based on the role or domain you're targeting. A partially finished, honestly-scoped-down capstone is a stronger portfolio piece than an over-claimed one that doesn't hold up when run.
