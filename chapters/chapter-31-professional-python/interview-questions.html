Chapter 31 Interview Prep 55 min read

Interview Questions: Professional Python

Writing Python that runs on your machine and shipping Python other people can install, run, and debug are different skills — this is where interviewers start probing for the second one. This set covers virtual environments, dependency management, why real tools use logging instead of print(), structuring a command-line interface with argparse, and what actually makes a folder of .py files an installable package.

1 Why do virtual environments matter? What actually goes wrong without one?
Strong answer

A virtual environment is an isolated Python installation with its own site-packages folder, separate from the system Python and from every other project's environment. Without one, every project on a machine installs its dependencies into the same global Python, which means two projects that need different versions of the same package (say, one needs requests==2.20 and another needs requests==2.31) can't both work at once — installing one silently breaks the other. It also means "works on my machine" stops being reproducible: someone else's global Python has a different accumulated pile of packages installed over time, so a project that quietly depended on some unlisted package being globally present breaks the moment it's cloned onto a clean machine.

Red flags

Describing venvs as just "a folder" with no mention of the actual failure mode (version conflicts between projects, or non-reproducible installs) they solve.

Follow-up

"If you `pip install` something without activating a venv first, where does it go?" (Into the global/system Python's site-packages -- exactly the shared space a venv exists to avoid touching, which is why it's easy to accidentally pollute a system Python by forgetting to activate first.)

2 What specifically can go wrong if you install packages globally instead of per-project?
Strong answer

Three concrete failure modes: version conflicts (two projects need incompatible versions of the same library, and only one can be installed globally at a time), silent hidden dependencies (a project runs fine locally because some unrelated tool already installed a package it happens to need, but that dependency is missing from requirements.txt and the project breaks for anyone else), and harder cleanup (uninstalling one project's dependencies risks breaking another project or the OS's own Python-based tooling, since some operating systems use system Python for their own scripts). All three are avoidable by giving every project its own venv and only ever installing into an activated one.

Red flags

Treating this as a purely hypothetical/academic risk rather than something that actually happens on real teams within weeks of skipping venvs.

Follow-up

"Some Linux distributions actively block `pip install` into the system Python now -- why?" (Because the OS itself relies on specific package versions for its own tooling; a stray global pip install can silently break OS-level scripts, so modern distros require an explicit override flag or a venv to prevent exactly this class of damage.)

3 What does requirements.txt actually do, and what are its limitations?
Strong answer

requirements.txt is a plain list of packages (optionally with version constraints) that pip install -r requirements.txt reads and installs in one command -- it's how a project communicates "here's everything you need to run this" to anyone who clones it. Its main limitation: an unpinned entry like requests (no version) installs whatever the latest version happens to be at install time, which can differ between two people installing on different days -- reintroducing the exact "works on my machine" problem venvs were supposed to fix, just one layer up. It also doesn't pin the transitive dependencies of your dependencies (the packages your packages depend on), so even a pinned top-level list can still resolve differently over time unless something more like a lockfile is used.

Red flags

Treating "has a requirements.txt" and "has reproducible installs" as the same thing -- an unpinned requirements.txt gives you neither reproducibility nor a record of what actually got installed and tested.

Follow-up

"What's the difference between requirements.txt and a lockfile (like poetry.lock or a pip-compile output)?" (requirements.txt is usually hand-written and loosely or fully unpinned; a lockfile is generated, pins every package -- including transitive dependencies -- to an exact version and often a hash, and is meant to be regenerated by tooling rather than edited by hand, giving byte-for-byte reproducible installs.)

4 Why does professional code use the logging module instead of scattering print() statements?
Strong answer

print() always writes to stdout, always shows every message, and has no concept of severity or where it should go -- every call has to be manually added or deleted to change verbosity. logging gives every message a severity level (DEBUG, INFO, WARNING, ERROR, CRITICAL), and one runtime setting (the logger's configured level) controls which of those actually get emitted -- so a tool can ship dozens of `logger.debug(...)` calls that stay silent by default and only appear when a user passes `--verbose`, with zero code changes. `logging` also supports multiple destinations at once (console, a log file, a remote log-aggregation service) and structured, consistent formatting (timestamps, logger name, level) applied automatically to every message -- none of which `print()` gives you without hand-rolling it yourself.

Red flags

Saying "logging is just fancier print()" -- the actual point is that severity and destination become configuration, not code you have to add/remove/comment out.

Follow-up

"Is `print()` ever still the right choice in a CLI tool?" (Yes -- for the tool's actual output, the result the user asked for, like `taskbox list` printing the task list itself. `logging` is for diagnostic/operational messages about what the program is doing internally; `print()` is fine for the program's actual product output that the user wants piped, redirected, or read directly.)

5 Walk through Python's standard log levels and when you'd use each.
Strong answer

From least to most severe: DEBUG (detailed internal state useful only while diagnosing a problem -- file paths being read, loop iteration counts, exact values being compared), INFO (confirmation that something expected happened -- "task added," "file saved," the normal narrative of a successful run), WARNING (something unexpected or worth noting, but the program can still continue -- "config file missing, using defaults," "task was already marked done"), ERROR (something failed and the current operation couldn't complete, but the program itself can keep running -- a single bad request in a loop of many), and CRITICAL (the program itself likely can't continue -- a required resource is completely unavailable). The default level if you don't configure anything is WARNING, meaning INFO and DEBUG are silent until explicitly enabled.

Red flags

Using ERROR or CRITICAL for routine, expected conditions (or the reverse -- logging a genuine failure at INFO where it'll be missed) -- the whole point of levels is that someone scanning only WARNING and above should see everything that actually needs attention.

Follow-up

"If you don't call logging.basicConfig() or attach any handler at all, what happens when you call logger.warning(...)?" (It still prints -- Python's logging module has a "handler of last resort" that outputs WARNING and above to stderr even with zero configuration, which is also exactly why the default level is WARNING and not something noisier.)

6 What's the difference between the root logger and a module-level logger (logging.getLogger(__name__))?
Strong answer

Every logger created with logging.getLogger("some.name") sits in a hierarchy and has a parent -- by default, every named logger's ultimate parent is the root logger (logging.getLogger(), no name). logging.basicConfig() configures the root logger specifically. Because log records propagate upward through the hierarchy by default (a child logger's message also flows to its parent's handlers, all the way to root), configuring the root logger affects every module and every imported library that uses logging.getLogger(__name__) without its own separate configuration -- not just your own code. Using logging.getLogger("yourtool") and attaching a handler directly to that named logger (with propagate = False if you don't want it also hitting root's handlers) scopes your configuration to your own tool's messages only.

Red flags

Not knowing that basicConfig() configures root, not "logging in general" -- this is the single most common real-world logging mistake, and it silently affects every dependency's log output, not just your own.

Follow-up

"You call basicConfig(level=logging.DEBUG) in your CLI tool. A library you imported also uses logging.getLogger(__name__) with no config of its own. What happens when your tool runs?" (That library's DEBUG-level messages print too, mixed in with yours -- confirmed by actually running this: a fake dependency's internal "opened connection pool" / "retry loop" debug lines appeared right alongside the tool's own debug output, with no way to separate them without editing code, since both loggers propagate to the same globally-configured root.)

7 In argparse, what's the difference between a positional and an optional argument, and how do you decide which to use?
Strong answer

A positional argument (parser.add_argument("title") -- no leading dashes) is identified by its position in the command and is required by default; leaving it out is an argparse error. An optional argument (parser.add_argument("--priority") -- leading dashes) can appear anywhere, in any order, and is optional by default unless you explicitly pass required=True. The decision is about whether the command means anything without it: a task's title is meaningless to omit (there's no sensible default for "what is this task called"), so it should be positional and required; a task's priority has a sensible fallback ("normal"), so it belongs as an optional flag with a default. Making something optional with a plausible-looking default when it should actually be required lets a user accidentally omit critical information with zero warning.

Red flags

Treating the choice as purely stylistic rather than a real correctness decision -- an optional argument that should have been required fails silently (wrong default used) instead of loudly (argparse's own clear error), which is a genuinely different user experience.

Follow-up

"Can an optional-style argument (with dashes) still be required?" (Yes -- add_argument("--priority", required=True) makes a dashed flag mandatory. It still has to be passed with its flag name (order doesn't matter), unlike a positional, but omitting it is now an argparse error instead of silently falling back to a default.)

8 How do you add subcommands to an argparse CLI (like `git commit` or `taskbox add`)?
Strong answer

parser.add_subparsers(dest="command", required=True) creates a subparsers object; calling .add_parser("add") on it returns a brand-new, independent ArgumentParser scoped to just that subcommand, with its own positional/optional arguments that don't leak into other subcommands. dest="command" stores which subcommand name was chosen into args.command, so the main function can branch on it (if args.command == "add": ...). required=True makes choosing a subcommand mandatory -- without it, running the program with no subcommand at all silently does nothing instead of showing a clear "you must pick one" error.

Red flags

Forgetting required=True and not noticing the tool silently no-ops when no subcommand is given -- a real usability bug that's easy to miss during casual testing since "run with valid args" is what gets tested most.

Follow-up

"Can a top-level flag like --verbose be shared across all subcommands without repeating add_argument for each one?" (Yes -- add it to the top-level parser before creating subparsers; argparse requires it to appear before the subcommand name on the command line (`tool --verbose add ...`, not `tool add --verbose ...`) unless you also add it individually to each subparser.)

9 What does pyproject.toml replace, and why did Python packaging move to it?
Strong answer

pyproject.toml replaces the legacy combination of setup.py (an executable Python script that had to run arbitrary code just to report a package's metadata -- a real security and reproducibility problem, since installing a package meant executing its author's code before you'd even decided to trust it) plus setup.cfg for static config. pyproject.toml is a static, declarative TOML file -- a build tool can read a package's name, version, and dependencies without executing anything. It's also become the single shared config file for the whole Python tooling ecosystem beyond just packaging (pytest, black, ruff, mypy, and others all read their own settings from `pyproject.toml` sections), rather than every tool inventing its own separate config file format.

Red flags

Claiming setup.py is entirely gone/unsupported -- some projects (especially older ones, or ones needing genuinely dynamic build logic) still use it or a hybrid approach; pyproject.toml is the modern standard and default, not a hard requirement enforced everywhere yet.

Follow-up

"Does pyproject.toml eliminate setup.py entirely, or can they coexist?" (They can coexist -- a project can have both a pyproject.toml (for standard metadata/build-system declaration) and a setup.py (for genuinely dynamic build steps that can't be expressed declaratively), though for most projects pyproject.toml alone is sufficient and setup.py isn't needed at all.)

10 What actually makes a folder of Python files "pip-installable"?
Strong answer

Three things together: (1) a pyproject.toml declaring a [build-system] (which build backend, like setuptools, should turn the source into an installable distribution) and a [project] table with at minimum a name and version; (2) the actual package -- a directory containing an __init__.py file, which is what makes Python treat that directory as an importable package rather than just a folder of loose scripts; (3) enough structure for the build backend to find that package -- either a conventional layout it auto-detects, or an explicit [tool.setuptools] packages = [...] declaration. Missing any of the three produces a real, reproducible failure: no `[project]` table with a `version` fails the build outright with a configuration error; no `__init__.py` means nothing is importable even if the install technically succeeds.

Red flags

Saying "just add a pyproject.toml" without mentioning that a genuinely missing required field (like `version`, with no `dynamic` declaration either) fails the build -- confirmed by actually running `pip install -e .` against a `pyproject.toml` missing `version`: it fails with `ValueError: invalid pyproject.toml config: 'project'. configuration error: 'project' must contain ['version'] properties`, thrown from a build subprocess several layers deep, not from pip's own top-level error handling.

Follow-up

"What's the difference between `pip install .` and `pip install -e .`?" (Plain `pip install .` copies the package into site-packages as a snapshot -- editing the source afterward has no effect until you reinstall. `pip install -e .` (editable/development install) installs a link back to the source directory instead, so changes to the `.py` files take effect immediately on the next run, no reinstall needed -- the standard choice while actively developing a package.)

11 What is a console_scripts entry point ([project.scripts] in pyproject.toml), and what does it actually do at install time?
Strong answer

[project.scripts] maps a command name to a Python function, in the form command-name = "module.path:function_name" -- e.g. taskbox = "taskbox.cli:main". At install time, pip reads that mapping and generates a small executable wrapper script (placed in the environment's `bin`/`Scripts` folder, which is on `PATH` while the venv is active) that imports the given module and calls the given function. That's the entire mechanism behind being able to type `taskbox` in a terminal instead of `python3 -m taskbox.cli` or `python3 path/to/cli.py` -- it's the exact same mechanism that gives you the `pip`, `black`, and `pytest` commands after installing those packages.

Red flags

Thinking the command name has to match the package name -- it doesn't; `[project.scripts]` can define multiple entries pointing at the same function (a short alias plus a full name), or a command name entirely different from the package.

Follow-up

"If you uninstall the package with pip uninstall, does the console_scripts command disappear too?" (Yes -- the generated wrapper script is tracked as part of the package's installed files, so uninstalling the package removes the command along with everything else pip installed for it.)

12 Why does proper argument handling and help text turn a script into a real "tool"?
Strong answer

A hand-rolled script that reads sys.argv[1], sys.argv[2] directly has no self-documentation (the only way to learn its interface is to read the source), no validation (a typo'd argument either crashes with a raw traceback or silently does the wrong thing), and no consistent conventions (every script invents its own argument order and error style). argparse gives every tool a free, auto-generated --help screen, automatic type conversion and validation (rejecting bad input before your code ever runs, with a clear message naming exactly what was wrong), and a consistent, familiar interface shape that matches every other real CLI tool a user has ever touched. That gap -- self-documenting, validated, predictable vs. "read the source and hope" -- is the actual line between a script and a tool other people (including future you) can pick up without help.

Red flags

Treating --help as a nice-to-have polish item rather than the main artifact that makes a tool usable by anyone who isn't its author.

Follow-up

"Does argparse's auto-generated --help update itself if you add a new subcommand later?" (Yes -- the help text is generated from the parser's current configuration (its arguments, their help= strings, choices, defaults) every time --help runs, so it's never able to drift out of sync with the actual accepted arguments the way a hand-written help string or README section can.)

13 What are common CLI UX mistakes that make a tool frustrating to use, even if it technically works?
Strong answer

A few recurring ones: no --help at all, or help text that doesn't explain what each argument actually does (just restating its name); unclear error messages -- a raw Python traceback instead of a plain-language explanation of what went wrong and how to fix it (compare a raw `KeyError: 999` traceback to `ERROR: No task with id 999.`); exit codes that don't reflect success/failure, so the tool can't be reliably used in a shell script that checks `$?`; silently accepting bad input instead of rejecting it (the "optional argument that should have been required" mistake -- a missing required value quietly becomes a wrong default instead of a clear error); and inconsistent flag conventions across subcommands (e.g. `--file` meaning something different, or being spelled differently, between two subcommands of the same tool).

Red flags

Only listing "no --help" as the one mistake -- that's the most obvious one, but silent-wrong-default and leaking raw tracebacks to end users are the ones that cause real damage (bad data, confused users) rather than just mild annoyance.

Follow-up

"Why does exit code matter if a human is just reading the terminal output anyway?" (Because CLI tools frequently get called from other programs -- shell scripts, CI pipelines, cron jobs -- that check the numeric exit code, not the printed text, to decide whether something succeeded. A tool that always exits 0 even on failure will make an automated pipeline silently continue past a real error.)

14 In this chapter's taskbox project, why is the logic split into core.py and cli.py instead of one file, and why does --file matter for testing?
Strong answer

core.py holds the actual task logic (loading/saving JSON, adding/listing/completing/deleting tasks) and never imports argparse, never touches sys.argv, and never calls print() -- it raises a custom TaskboxError exception on bad input instead. cli.py is the only place that parses arguments, configures logging, and prints to the terminal; it calls into `core.py`'s functions and translates their return values (or a caught `TaskboxError`) into terminal output. That separation means `core.py` is directly unit-testable (call `add_task()` with a scratch file path and assert on the returned dict, no terminal involved) and could be reused by a future GUI or web API without dragging any command-line concerns along with it. The top-level `--file` flag (rather than a hardcoded `~/.taskbox/tasks.json` path baked into the functions) is what makes that testability real in practice -- every test, and every manual verification run, can point at an isolated scratch file instead of touching (or depending on the state of) the user's real task list.

Red flags

Not being able to explain the split beyond "it's cleaner" -- the concrete payoff (testability without a terminal, reusability outside a CLI context) is the actual answer an interviewer is listening for.

Follow-up

"If core.py needs to report something like 'this task was already marked done' -- not fatal, just worth noting -- should it print, raise, or log?" (Log it, at WARNING level, via `logger.warning(...)` -- it's not an error (the operation still succeeds), it's not the tool's actual output (so not print()), and core.py shouldn't be deciding what the terminal displays at all -- that's cli.py's job, which is exactly why core.py logs instead of printing.)
