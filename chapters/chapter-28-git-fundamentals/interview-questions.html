Chapter 28 Interview Prep 55 min read

Interview Questions: Git Fundamentals

Every software job assumes you already know git — so interviewers rarely ask "what is git," they ask questions that reveal whether you've actually broken something and fixed it. This set covers merge vs. rebase, the staging area, conflict resolution, resetting safely, stashing, squashing, and the safety nets (reflog, blame, bisect) that separate someone who's memorized commands from someone who can recover from a real mistake.

1 What's the difference between git merge and git rebase, and when would you use each?
Strong answer

Both combine changes from one branch into another, but they produce different history. git merge creates a new merge commit that ties both branches' histories together exactly as they happened — nothing is rewritten, so it's always safe, even on shared branches. git rebase instead replays your branch's commits one by one on top of the target branch's latest commit, producing a straight line of new commits with new hashes — cleaner-looking history, but it rewrites commits that already existed. Use merge for anything shared (bringing a feature branch into main) since it preserves true history and is non-destructive; use rebase to clean up your own not-yet-shared feature branch before opening a pull request, or to catch it up with a moved-on main without an extra merge commit.

Red flags

Saying one is "just better" than the other with no mention of shared vs. local history — that distinction is the entire point of the question.

Follow-up

"If two people are already working off your feature branch, is it safe to rebase it?" (No — rebasing rewrites commit hashes, so anyone who already pulled the old commits now has history that's diverged from yours; their next pull or push gets messy. Don't rebase a branch other people have already based work on.)

2 What is a "detached HEAD" state, and how do you get out of it safely?
Strong answer

Normally HEAD points at a branch name (like main), which in turn points at a commit — so committing moves the branch pointer forward along with HEAD. Checking out a specific commit hash directly (git checkout <sha>) instead of a branch name points HEAD straight at that commit, with no branch attached — that's detached HEAD. You can look around and even make experimental commits there, but since no branch name is tracking them, they become unreachable and eligible for garbage collection the moment you switch to another branch, unless you first run git switch -c new-branch-name to give them a real branch to live on.

Red flags

Treating detached HEAD as an error state to panic about, rather than understanding it's a normal (if easy to end up in by accident) part of how HEAD works.

Follow-up

"You made 3 commits in a detached HEAD, then ran git checkout main — are those commits gone?" (Not immediately — they still exist and are recoverable through git reflog until git's garbage collector eventually cleans them up, but they're not reachable from any branch until you create one pointing at them.)

3 Walk through the difference between git reset --soft, --mixed, and --hard.
Strong answer

All three move the current branch pointer (and HEAD) to a different commit — they differ only in what happens to the staging area and working directory along the way. --soft moves the branch pointer but leaves everything else untouched: your changes end up staged, ready to re-commit differently. --mixed (the default if you don't pass a flag) also unstages those changes, leaving them as unstaged edits in your working directory. --hard goes furthest: it moves the pointer and overwrites your staging area and working directory to match the target commit exactly, discarding any uncommitted work with no recovery path other than the reflog.

Red flags

Not knowing that --hard is the only one of the three that can destroy uncommitted work, or recommending it casually without that warning.

Follow-up

"You have uncommitted changes you care about and you want to undo your last commit but keep those changes staged — which flag?" (--soft — it moves HEAD back one commit but leaves the commit's changes staged, ready to fold into a new commit.)

4 How does git represent a merge conflict inside a file, and how do you resolve one?
Strong answer

When the same region of a file was changed differently on both branches, git can't auto-merge it, so it writes both versions directly into the file with conflict markers: <<<<<<< HEAD starts your current branch's version, ======= separates it from the other branch's version, and >>>>>>> branch-name marks the end. Resolving means opening the file, deciding what the final content should actually be (keep one side, keep both, or write something new entirely), deleting all three marker lines, then running git add on the file to mark it resolved and git commit (or git merge --continue) to finish the merge.

Red flags

Forgetting that git add is required after editing the file — without it, git still considers the path unresolved even if the markers are gone.

Follow-up

"If you decide the merge was a mistake entirely, how do you back out?" (git merge --abort — restores the working directory to exactly how it was right before the merge attempt started, as long as you haven't committed yet.)

5 What is the staging area (the "index"), and why does git have one instead of just committing the working directory directly?
Strong answer

The staging area is an intermediate snapshot between your working directory and your next commit — git add copies a file's current state into it, and git commit records whatever is in the staging area at that moment, not necessarily every change sitting in your working directory. That separation lets you build a commit deliberately: stage only some of the files you've changed (or even part of one file with git add -p) so a commit represents one coherent change, instead of being forced to commit literally everything you've touched at once.

Red flags

Describing git add as just "saving" a file, with no mention of the staging area being a distinct, inspectable state from either the working directory or the last commit.

Follow-up

"What's the difference between git diff and git diff --staged?" (git diff shows unstaged changes — working directory vs. the staging area. git diff --staged (or --cached) shows staged changes — staging area vs. the last commit. They're comparing different pairs.)

6 What does git stash do, and when would you reach for it?
Strong answer

git stash takes your current uncommitted changes (staged and unstaged) and saves them onto a stack, then resets your working directory back to match the last commit — as if the changes never happened, without actually committing or losing them. It's the tool for "I need a clean working directory right now" mid-task: switching branches to fix an urgent bug, pulling in upstream changes that would conflict with your in-progress edits, or just wanting a checkpoint before trying something risky. git stash pop reapplies the most recent stash and removes it from the stack; git stash list shows everything currently stashed, since you can stash more than once.

Red flags

Confusing stash with a commit — a stash isn't part of branch history and isn't pushed anywhere; it's local, working-directory-only state.

Follow-up

"What's the difference between git stash pop and git stash apply?" (Both reapply a stash's changes to your working directory, but pop also removes it from the stash stack afterward, while apply leaves it there in case you want to apply the same stash again elsewhere.)

7 What does "squashing commits" mean, and why would you do it?
Strong answer

Squashing combines several existing commits into one, discarding the individual boundaries between them while keeping their combined changes. It's most often done with an interactive rebase (git rebase -i) — marking commits after the first as "squash" (or "fixup" to also drop their commit messages) folds them into the commit above. The usual reason is cleanup before merging: a feature branch might accumulate a dozen "wip," "fix typo," "actually fix it" commits during development that are useful to you locally but noise to anyone reading history later — squashing them into one well-described commit makes the eventual main-branch history readable.

Red flags

Suggesting squashing commits that have already been pushed and are shared with teammates — same shared-history problem as rebase in general, since squashing is a form of rebase.

Follow-up

"What's the difference between 'squash' and 'fixup' in an interactive rebase?" (squash folds a commit into the one above and lets you edit the combined commit message; fixup does the same fold but silently discards the folded commit's message entirely, keeping only the one it merges into.)

8 How would you use git blame and git bisect to debug a production issue?
Strong answer

git blame file.py annotates every line of a file with the commit, author, and date that last changed it — the fastest way to answer "who wrote this line and why" when you've already found the suspicious code and just need context (often leading you to the commit message or the linked ticket). git bisect is for when you don't know which commit caused a regression at all: you tell it a known-good commit and a known-bad one, and it binary-searches the commits between them, asking you at each step to mark that commit good or bad (or automate the check with a script via git bisect run) — narrowing hundreds of commits down to the exact one that introduced the bug in roughly log2(n) steps instead of checking every commit one by one.

Red flags

Only knowing blame and not bisect (or vice versa) — they solve different problems: blame explains a known line, bisect finds an unknown commit.

Follow-up

"Why is bisect a binary search rather than checking commits one at a time from newest to oldest?" (Binary search cuts the remaining range in half every step, so even a range of 1,000 commits resolves in about 10 checks instead of up to 1,000 — critical when each check means actually building and testing the code at that commit.)

9 Why is rewriting public/shared git history considered dangerous?
Strong answer

Operations like rebase, commit --amend, or a forced push (git push --force) don't edit existing commits in place — they create new commits with new hashes and move a branch pointer to them, effectively replacing the old commits. If anyone else has already pulled the old commits, their local history now points to commits that no longer exist on the branch's history going forward; their next pull creates a confusing divergent/duplicated history, or their next push can silently overwrite the rewritten work if they force-push back. The damage isn't to the rewriter — it's to everyone downstream who built on the version that just got erased.

Red flags

Treating git push --force as a routine fix for "my push was rejected" without first understanding why it was rejected (usually: someone else already pushed commits you don't have locally).

Follow-up

"Is there a safer alternative to a plain force push?" (git push --force-with-lease — refuses the push if the remote branch has commits you haven't seen locally, which plain --force pushes right over. It's not risk-free, but it catches the most common "someone else pushed in the meantime" accident.)

10 What's the difference between a fast-forward merge and a three-way merge?
Strong answer

A fast-forward merge happens when the branch you're merging in is simply ahead of your current branch, with no new commits on your side since they diverged — git can just move your branch pointer forward to match, with no new commit created and no possibility of a conflict. A three-way merge happens when both branches have new commits since they diverged — git has to look at three points (the common ancestor, and the tip of each branch) to figure out what changed on each side and combine them, producing a new merge commit with two parents. Conflicts are only possible in the three-way case, since fast-forward has nothing to reconcile.

Red flags

Assuming every merge produces a merge commit — a fast-forward merge deliberately doesn't, unless you explicitly pass --no-ff to force one for history-readability reasons.

Follow-up

"Why might a team pass --no-ff even when a fast-forward is possible?" (To always leave a visible merge commit marking where a feature branch was integrated, even when the history would technically allow a silent fast-forward — makes it easier to see "this group of commits was one feature" later in the log.)

11 What is the reflog, and how is it a safety net for mistakes like a bad reset --hard?
Strong answer

The reflog (git reflog) is a local, per-repository log of every place HEAD has pointed — every commit, checkout, reset, rebase, and merge that moved it, each with a timestamp and a short description, most recent first. Commits don't actually vanish the instant they become unreachable from any branch (say, after a reset --hard to an earlier commit) — they just sit unreferenced until garbage collection eventually cleans them up, which by default doesn't happen for a while. Since the reflog still lists the commit hash from before the reset, git reset --hard <that-hash> (found via git reflog) restores exactly that state, recovering commits that look "lost" from a normal git log.

Red flags

Not knowing the reflog exists at all, or suggesting a lost commit after reset --hard is unrecoverable.

Follow-up

"Does the reflog help if you deleted a branch on a teammate's machine, not your own?" (No — the reflog is entirely local, not pushed or shared. It only tracks HEAD movements in your own repository, so it can't recover something that only ever existed in someone else's local reflog.)

12 What does a .gitignore file do, and what's a common mistake people make with it?
Strong answer

.gitignore lists file and directory patterns that git should never track or show as "untracked" in git status — the standard place for build artifacts, dependency folders (node_modules, __pycache__), local environment files (.env), and IDE-specific settings that shouldn't be part of the shared repository. The common mistake: adding a pattern to .gitignore after a matching file has already been committed does nothing — .gitignore only affects untracked files. A file git is already tracking has to be explicitly removed from tracking too (git rm --cached <file>) before the ignore rule takes effect on it going forward.

Red flags

Assuming adding something to .gitignore retroactively removes it from history or from being tracked — it doesn't do either.

Follow-up

"You accidentally committed a .env file with a real secret in it. Is adding .env to .gitignore now enough?" (No — the secret is still in git history in every earlier commit, readable by anyone with access to the repo/its history. It needs actual history rewriting (or, more realistically, treating the secret as compromised and rotating it) — .gitignore only prevents the mistake from happening again going forward.)

13 What's the difference between a local branch and a remote-tracking branch (like origin/main)?
Strong answer

A local branch (main) is a pointer you can move directly — every commit, merge, or rebase on it updates that pointer immediately. A remote-tracking branch (origin/main) is a read-only bookmark recording where that branch was on the remote the last time you talked to it (via fetch, pull, or push) — it only updates when you actually communicate with the remote, not automatically as time passes. This is exactly why local main can drift ahead of or behind origin/main: someone else can push new commits to the real remote main, but your origin/main won't reflect that until you git fetch, even though the branch names look identical.

Red flags

Assuming origin/main is always in sync with the actual remote — it's a cached snapshot from your last fetch, not a live view.

Follow-up

"What's the difference between git fetch and git pull?" (git fetch updates your remote-tracking branches (origin/main) with what's actually on the remote, without touching your local branches or working directory at all. git pull does that fetch and then immediately merges (or rebases, with --rebase) the result into your current local branch — fetch is the safe "just look" step, pull adds the "and now apply it" step.)

14 What does git cherry-pick do, and when is it the right tool?
Strong answer

git cherry-pick <commit-hash> applies the changes from one specific commit somewhere else onto your current branch, creating a new commit with the same changes (and message, by default) but a new hash and a new parent — without bringing along any of that commit's surrounding history. It's the right tool when you need exactly one fix from another branch without merging or rebasing everything else on it — the classic case is a hotfix: a bug got fixed on a feature branch that isn't ready to merge yet, but the fix itself needs to land on main right away, so you cherry-pick just that one commit onto main instead of waiting for the whole feature.

Red flags

Reaching for cherry-pick as a routine way to move work between branches instead of a normal merge — cherry-picking many commits from the same branch tends to create duplicate, conflicting history once that branch eventually does get merged too.

Follow-up

"Can a cherry-pick conflict, the same way a merge can?" (Yes — if the target branch has diverged enough that the surrounding code the commit's diff expects to find isn't there anymore, git presents the same conflict markers and resolution process as a merge conflict.)
