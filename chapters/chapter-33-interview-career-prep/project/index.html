careerprep: a Mock-Interview Drill Tool + Portfolio Checklist Scanner

A real, multi-file, pip-installable command-line tool with two genuinely useful features for wrapping up this course: a mock-interview drill that saves your typed answers with a timestamp so you can watch them improve over multiple practice sessions, and a portfolio-readiness scanner that actually points at one of your own project folders and checks real files on disk — not a static list you tick by hand.

What you're building

Feature 1 — mock-interview drill (careerprep interview ...)
  • careerprep interview list [--category behavioral|technical] — list all 25 questions in the built-in bank (10 behavioral STAR-method questions, 15 technical-concept questions spanning Python fundamentals, OOP, testing, git, APIs, databases, memory, and concurrency — pulled from topics across this course).
  • careerprep interview random [--category behavioral|technical] — pick one random question, print it, read a typed/pasted multi-line answer from the terminal, and save it to a local JSON file with a timestamp.
  • careerprep interview review [--category ...] [--question-id N] — show previously saved answers, optionally filtered, so you can watch how your answers to the same question evolve across practice sessions. Every session appends — nothing is ever silently overwritten.
Feature 2 — portfolio checklist scanner (careerprep checklist ...)
  • careerprep checklist list — print the full 9-item checklist: 4 auto-checkable items and 5 manual self-assessment items.
  • careerprep checklist scan <path> — point it at a real directory (one of your own capstone project folders) and it actually checks 4 items for real: a non-trivial README.md, a requirements.txt/pyproject.toml, a test file (tests/ or test_*.py/*_test.py), and a .gitignore. The other 5 items (README explains WHY not just WHAT, no hardcoded secrets, an honest limitations section, tests exercise real logic, meaningful commit history) print as manual checkboxes — a script can't honestly judge those, so it doesn't pretend to.

A top-level --file <path> flag points interview commands at which JSON answers file to read/write (defaults to ~/.careerprep/answers.json). A top-level -v/--verbose flag raises the tool's own log level from INFO to DEBUG, scoped to careerprep's own logger only.

Why two modules, not one script interview.py and checklist.py hold the actual logic — reading/writing JSON, filtering questions, scanning a directory's real files. Neither imports argparse, touches sys.argv, or calls print(). cli.py holds the argparse parser, the input() calls that collect a typed answer, and is the only place that talks to the terminal. This is the same "separate the engine from the dashboard" split Chapter 31's taskbox project used.

Build it: interview.py

1 Filter and pick questions (TODOs 1-3)

get_questions validates the category and filters the built-in 25-question bank. pick_random_question reuses it and calls random.choice, raising CareerPrepError if a category has no matches (it never will, with this bank, but a learner adding new categories later would want that guard).

careerprep/interview.py (solution) 
def pick_random_question(category: str | None = None) -> dict:
    """Pick one random question, optionally restricted to a category."""
    questions = get_questions(category)
    if not questions:
        raise CareerPrepError(f"No questions available for category {category!r}.")
    question = random.choice(questions)
    logger.debug("Picked random question #%d (category=%s)", question["id"], question["category"])
    return question

Thought process: the empty-result case has to be handled before calling random.choice, which raises a confusing IndexError on an empty sequence rather than a message that tells the caller what actually went wrong. Filtering first through the already-validated get_questions also means the category-validation logic lives in exactly one place, not duplicated here.

2 Load, save, and append answers (TODOs 4-9)

load_answers/save_answers mirror Chapter 31's load_tasks/save_tasks pattern exactly — missing file is a normal empty start, unreadable/corrupted file gets a clear wrapped error. The one detail that matters most here is record_answer: it must load the existing answers and append, never replace the file with just the new entry — losing past practice sessions defeats the entire point of a drill tool meant to show improvement over time.

careerprep/interview.py (solution) 
def record_answer(answers_file: Path, question: dict, answer_text: str) -> dict:
    """
    Append a new answer to the answers file (never overwrite past ones --
    each practice session should add to the learner's history, not erase
    it, so they can see how their answers evolve over time).
    """
    answer_text = answer_text.strip()
    if not answer_text:
        raise CareerPrepError("Answer cannot be empty.")

    answers = load_answers(answers_file)
    entry = {
        "question_id": question["id"],
        "category": question["category"],
        "question_text": question["text"],
        "answer": answer_text,
        "answered_at": _now_iso(),
    }
    answers.append(entry)
    save_answers(answers_file, answers)
    logger.info("Saved answer for question #%d (%s)", question["id"], question["category"])
    return entry

Build it: checklist.py

3 Scan a real directory for objectively-checkable signals (TODOs 11-12)

scan_directory is the only function in this project that touches a real filesystem path a learner supplies. It checks four things a script genuinely can verify without judgment: does a README exist and clear a minimum length, does a dependency file exist, does a test file exist, does a .gitignore exist. Five more checklist items (README explains WHY, no hardcoded secrets, an honest limitations section, tests exercise real logic, meaningful commits) stay in manual_items() — a script that claimed to verify those would be lying about what it can actually see.

careerprep/checklist.py (solution) 
def scan_directory(target: Path) -> dict[str, bool]:
    if not target.exists():
        raise CareerPrepError(f"No such directory: {target}")
    if not target.is_dir():
        raise CareerPrepError(f"Not a directory: {target}")

    logger.debug("Scanning %s for portfolio-readiness signals", target)

    readme = _find_file(target, README_FILENAMES)
    results = {
        "readme_substantial": readme is not None and _is_substantial(readme),
        "dependencies_declared": any((target / name).is_file() for name in DEPENDENCY_FILENAMES),
        "tests_exist": _has_tests(target),
        "gitignore_exists": (target / ".gitignore").is_file(),
    }
    logger.info("Scanned %s: %d/%d auto-checkable items passed", target, sum(results.values()), len(results))
    return results

Thought process: "does a README exist" and "is the README actually useful" are two very different questions — the first is a file-existence check, the second requires judgment a script can't make honestly. The temptation with an AI assistant is to let it collapse "exists" into "passes," which quietly overstates what the scanner actually verified (this is exactly the flaw planted in this chapter's AI-paired version — go compare after you've built this).

4 Build the argparse parser with nested subcommands (TODOs 11-17)

This parser nests one level deeper than Chapter 31's taskbox: interview and checklist are each their own subparser with their own subparsers underneath (list/random/review, and list/scan). args.command tells you which top-level group ran; args.interview_command or args.checklist_command tells you which one inside that group.

careerprep/cli.py (solution) 
subparsers = parser.add_subparsers(dest="command", required=True)

interview_parser = subparsers.add_parser("interview", help="mock-interview drill tool")
interview_sub = interview_parser.add_subparsers(dest="interview_command", required=True)

random_parser = interview_sub.add_parser("random", help="answer one random question")
random_parser.add_argument(
    "--category", choices=["behavioral", "technical"], default=None,
    help="only pick from this category",
)

checklist_parser = subparsers.add_parser("checklist", help="portfolio-readiness checklist")
checklist_sub = checklist_parser.add_subparsers(dest="checklist_command", required=True)

scan_parser = checklist_sub.add_parser("scan", help="scan a real directory and auto-check what's checkable")
scan_parser.add_argument("path", type=Path, help="path to the project directory to scan")

Thought process: two-level subcommands (careerprep interview random) need required=True on both levels of add_subparsers, not just the top one — forgetting it on the inner level means careerprep interview with no further argument silently does nothing instead of showing a clear "missing command" error.

5 Scoped logging, multi-line input, and error handling (TODOs 18-25)

configure_logging reuses Chapter 31's named-logger pattern (never logging.basicConfig()). read_multiline_answer is new for this chapter: it loops on input(), collecting lines until a blank one is entered, so a learner can write a real multi-paragraph STAR answer instead of one cramped line. main() wraps dispatch in try/except (InterviewError, ChecklistError) so a bad path or bad category becomes a clean one-line error, not a raw traceback.

Why requirements.txt is (correctly) empty careerprep only imports from the standard library: argparse, logging, json, pathlib, datetime, random, textwrap. There is nothing to pin. The file is kept — not deleted — with a comment explaining that.

Why this project matters

Every real interview loop combines exactly these two things: practicing how you talk about your work out loud, and having a portfolio that actually holds up when someone opens the folder. A checklist you never run against real files is just a list you feel good about; a mock-interview tool that doesn't save your answers gives you no way to notice you're getting better. This project is a small, honest version of both — the same shape you'd want for real job prep, built with the same subcommands-plus-package pattern you used for taskbox back in Chapter 31.

More project ideas

No starter/solution files are provided for these on purpose — building and verifying each one yourself is the point. Each is solvable with only what this chapter (and earlier chapters) have taught.

1 Add a --timer flag that times how long you take to answer each question

Scenario: Real interviews are timed, even if informally — a behavioral answer that takes eight minutes to get to the point is a real problem, and you won't notice unless you measure it.

What to do: Add a --timer flag to interview random. When set, start a timer (time.monotonic()) right after the question is printed, and stop it right after read_multiline_answer() returns. Print the elapsed time, and save it alongside the answer in the JSON entry (a new elapsed_seconds key).

Suggested approach: time.monotonic() before and after, subtract, round(..., 1) for a clean display — don't use time.time(), which can jump backward if the system clock adjusts mid-interview.

2 Add a 1-5 confidence self-rating after each answer

Scenario: Raw answer text doesn't tell you which questions you actually feel shaky on — a self-rating, tracked over time, does.

What to do: After saving an answer in interview random, prompt "How confident do you feel about that answer? (1-5): ", validate it's an int 1-5 (re-prompt on bad input), and save it in the JSON entry alongside the answer as a confidence key. Add a way to see your average confidence per question when reviewing.

Suggested approach: A small while True: input-validation loop, the same pattern you used in earlier chapters' menu-driven projects.

3 Extend the checklist scanner to check for a LICENSE file and a CI workflow

Scenario: Two more genuinely checkable portfolio signals: does the repo have an open-source license, and does it have any automated CI at all.

What to do: Add two more auto-checkable items to CHECKLIST_ITEMS: license_exists (a LICENSE, LICENSE.md, or LICENSE.txt file at the project root) and ci_workflow_exists (a .github/workflows/ folder containing at least one .yml/.yaml file). Wire both into scan_directory()'s results dict.

Suggested approach: Reuse the _find_file() helper pattern for LICENSE*; for the CI check, (target / ".github" / "workflows").is_dir() and any((target / ".github" / "workflows").glob("*.y*ml")).

4 Add an export-to-markdown feature that turns saved answers into a study guide

Scenario: A JSON file of past answers isn't something you'd actually review the night before an interview — a readable document is.

What to do: Add a new careerprep interview export <output.md> subcommand. It should load all saved answers (optionally filtered by --category), group them by question, and write a Markdown file with one ## Question text heading per question followed by each of your saved answers to it in chronological order.

Suggested approach: Build a dict[int, list[dict]] keyed by question_id from the loaded answers, then iterate it to build the Markdown string before writing it with Path(output).write_text(...).
