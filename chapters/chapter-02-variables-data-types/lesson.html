Variables & Data Types

You already used variables in Chapter 1. Now let's understand exactly what's happening under the hood.

On this page 1. Naming variables properly 2. Python decides the type for you 3. type() and id() 4. Converting between types
Every value your program touches — a name, a price, a score — lives somewhere in memory with a specific type and a specific address. Most of the time Python hides this from you. But the moment something goes wrong (a crash, a weird bug, an interview question you can't answer), it's because one of these hidden details mattered. This chapter makes them visible.
Sub-topic 1 of 4

1. Naming variables properly

In Chapter 1 you wrote name = "Asha" without worrying too much about what names are allowed. Python actually has firm rules about it — break them, and your program won't even run.

naming.py 
abc123 = "valid"      # starts with a letter — OK
_123 = "valid"        # starts with underscore — OK
123abc = "invalid"    # cannot start with a digit
abc 123 = "invalid"   # no spaces allowed
abc$123 = "invalid"   # no special symbols allowed
What is... case-sensitivity? Python treats uppercase and lowercase letters as completely different characters in a name. age, Age, and AGE are three separate variables. This trips up beginners constantly — if your code says a variable is "undefined" but you swear you created it, check the capitalization first.
Go Deeper: why these rules exist

Python's parser needs to instantly tell a variable name apart from a number or an operator while reading your code left to right. Letting names start with a digit would make 123abc ambiguous — is that the number 123 followed by something called abc, or one name? Banning that ambiguity up front keeps the language simple to parse and impossible to misread.

Quick recap
  • A variable name must start with a letter or underscore (_).
  • After that, digits are allowed.
  • No spaces, no special symbols ($, %, etc.).
  • Names are case-sensitive: age ≠ Age.
Sub-topic 2 of 4

2. Python decides the type for you

In languages like Java or C, you must announce a variable's type before using it. Python doesn't ask you to — it looks at the value you assign and figures the type out automatically. This is called dynamic typing, and it's one of the biggest reasons Python feels so much lighter to write.

auto_types.py 
name = "Asha"        # str    — text
age = 12             # int    — whole number
height = 4.9         # float  — decimal number
is_student = True    # bool   — True or False
reading = 3 + 4j     # complex — rarely needed, but built in

print(name, age, height, is_student, reading)

What is... a boolean? A boolean is a data type with exactly two possible values: True or False (capitalized, no quotes — those are keywords, not strings). Booleans are what every comparison like 5 > 3 produces, and they're the foundation of every decision your programs will make starting in Chapter 4.
What is... dynamic typing? Dynamic typing means a variable's type is determined automatically from the value assigned to it, and can even change later in the same program: x = 10 then later x = "ten" is completely legal — x just points to a different type of value now.
Go Deeper: complex numbers

A complex number has a real and an imaginary part, written as a+bj in Python (using j instead of math's usual i, a convention borrowed from electrical engineering). Most beginner programs never need this type, but it exists as a first-class built-in type — useful to know if you ever move into scientific computing.

Quick recap
  • Python is dynamically typed — no need to declare a type.
  • Core types: str, int, float, bool, complex.
  • True and False are booleans, not strings — no quotes.
Sub-topic 3 of 4

3. Peeking under the hood: type() and id()

Since Python decides types automatically, you sometimes need to ask it what it decided. Two built-in functions answer that: one tells you a value's type, the other tells you exactly where in memory it lives.

type_and_id.py 
x = 10
print(type(x))   # <class 'int'>
print(id(x))     # some memory address, e.g. 140713...

y = 10
print(id(y))     # often the SAME address as x!
What is... identity? An object's identity is its unique location in memory, returned by id(). Two variables can hold "the same value" while being different questions from "are they the exact same object in memory" — that second question is what id() and the is keyword answer.
Go Deeper: why do x and y share an address?

For small, unchanging values like the integer 10, Python reuses one object in memory instead of creating a new one every time. When you write x = 10 and y = 10, both variables just point at that one shared object — that's why id(x) == id(y). This works because numbers, strings, and booleans in Python are immutable: once created, an object's value can never change in place. If you later do x = 20, Python doesn't modify the original object — it creates a brand new one and points x at that instead, leaving y untouched at 10.

Quick recap
  • type(value) tells you the data type.
  • id(value) tells you the memory address.
  • Numbers, strings, and booleans are immutable — they can't be changed in place.
Sub-topic 4 of 4

4. Converting between types

Remember from Chapter 1: input() always returns a string, even if the person types a number. Sooner or later you need to explicitly convert a value from one type to another — Python gives you one function per type to do it.

conversion.py 
age_text = "25"
age_number = int(age_text)      # str -> int
print(age_number + 5)           # 30

price = float("19.99")          # str -> float
name_of_25 = str(25)            # int -> str
is_ready = bool(1)               # -> True

What is... type conversion? Type conversion (also called "casting") is turning a value of one type into another — int("25") turns the string "25" into the number 25. It only works if the value actually makes sense in the new type: int("hello") raises an error, because "hello" isn't a number in any language.
Go Deeper: the bool() trap that catches everyone once

bool("False") returns True. Read that again — it's not a typo. bool() doesn't look at what the text says, it only checks whether the string is empty or not. Any non-empty string, including the literal text "False", converts to True. Only bool("") (an empty string) gives False. This exact mistake shows up in real bugs when developers read a setting like "false" from a config file as text and forget this rule.

Quick recap
  • int(), float(), str(), bool(), complex() convert between types.
  • Conversion fails with an error if the value doesn't make sense in the target type.
  • bool() on a string only checks if it's empty — not what the text says.
In the Real World
  • Every "500 Internal Server Error" you've ever seen on a website is often, under the hood, a TypeError from mixing types the way this chapter warned about — unhandled by that site's code.
  • Config files (.env, JSON settings) load everything as strings or loosely-typed values first — real production code constantly does exactly the int()/float() conversions you just practiced before trusting that data.
  • id() and immutability aren't academic trivia — they're the exact reasoning behind why Python's small-integer caching exists, and why comparing objects with is vs == causes real, hard-to-find bugs in production systems.

Points to Remember

  • Variable names: start with a letter or underscore, then letters/digits only, no spaces or symbols, case-sensitive.
  • Python is dynamically typed — it infers str, int, float, bool, or complex from the value you assign.
  • type(x) shows the data type; id(x) shows the memory address.
  • Small immutable values (like small ints) may share the same memory address until reassigned.
  • Convert between types explicitly with int(), float(), str(), bool(), complex() — watch out for the bool()-on-strings trap.
