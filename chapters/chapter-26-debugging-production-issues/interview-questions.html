Interview Questions: Debugging Production Issues

Debugging questions in real interviews rarely ask you to name an exception in isolation — they show you a symptom or a snippet and ask you to diagnose it, the same way this chapter's catalog is organized. This set covers the ten crash patterns from Category 1: Crashes & Exceptions.

1 A dict lookup crashes with KeyError in production but never did in testing. Walk me through how you'd debug it.
Strong answer

Start from the traceback: it names the exact missing key and the exact line. The real question is why the key is missing in production but not in tests — usually a config/environment difference (an env var that's set locally but not in the deploy environment, a config file that isn't shipped the same way). The fix depends on whether the key is genuinely optional (use .get() with a sensible default) or required (fail fast at startup with a clear message naming every missing key, not just the first one hit).

Red flags

Jumping straight to "just use .get() everywhere" without asking whether the key is actually supposed to be required — that just turns a loud, diagnosable crash into a silent misconfiguration.

Follow-up

"What's the downside of wrapping every dict access in a try/except KeyError?" (It can hide genuine bugs the same way a bare except: does — the goal is deciding per-key whether missing is expected, not blanket-suppressing the error everywhere.)

2 You see IndexError: list index out of range in a loop that uses range(n). What's your first hypothesis?
Strong answer

range(n) produces indices with no awareness of any particular list's actual length — it's just a sequence of integers. The first hypothesis is a mismatch between n (however it was chosen) and len(list_being_indexed). I'd check whether n comes from a different source than the list itself (a separate count, a hardcoded assumption, a different collection's length) — that's almost always where the two silently drift apart.

Red flags

Suggesting a blanket try/except IndexError around the loop instead of fixing the actual bound — that hides the bug instead of preventing it.

Follow-up

"What's a bound that can never cause this bug?" (range(len(the_actual_list)), or just iterating the list directly instead of indexing by position at all.)

3 What's wrong with a bare except:, specifically — not "it's bad practice," but what concretely goes wrong?
Strong answer

It catches every exception, not just the one you had in mind — including ones that indicate a real, unrelated bug (a typo'd dict key, a NameError from a refactor, even KeyboardInterrupt/SystemExit in older code). The concrete failure mode: a function returns a fallback value that looks completely normal, while masking a bug that silently produces wrong behavior for months. The fix is always to name the specific exception you expect (except KeyError:), so anything else still surfaces as its own distinct, diagnosable crash.

Red flags

A vague "it's just bad style" answer with no concrete failure scenario — this needs a real example of what breaks.

Follow-up

"Is except Exception: just as bad?" (Nearly — it's slightly narrower than bare except: (it won't catch SystemExit/KeyboardInterrupt), but it's still broad enough to swallow bugs you didn't anticipate, same failure mode.)

4 A TypeError shows up three function calls deep from where the actual bad value entered the system. How do you find the real source?
Strong answer

Read the traceback bottom-to-top: the deepest frame shows where the crash actually happened (the arithmetic that failed), but the value that caused it was likely valid-looking all the way through several calls until it hit an operation that couldn't tolerate its type. I'd trace the value backward through each frame in the traceback to find where it first entered the call chain — usually external input (form data, a query parameter, an API response) that was never converted or validated at the boundary.

Red flags

Only looking at the last line of the traceback and proposing a type-check fix at that exact spot, rather than validating input at the actual entry point.

Follow-up

"Where's the right place to fix this — where it crashed, or where the bad value entered?" (Where it entered — fixing it at the crash site just moves the problem one call deeper next time.)

5 AttributeError: 'NoneType' object has no attribute 'X' is one of the most common production tracebacks. What's the usual root cause?
Strong answer

Almost always a function that returns None on some code path — often implicitly, by falling through without an explicit return — and a caller that assumes the return value is always a real object. The fix is either checking for None right where the value comes back, before calling anything on it, or having the source function raise a specific, named exception instead of quietly returning None when nothing was found.

Red flags

Suggesting a blanket if x: check without recognizing that "falsy but not None" values (empty list, 0, empty string) can trigger the same branch incorrectly if the check isn't specifically is None.

Follow-up

"Why is 'is None' preferred over just 'if not x' here?" (Because a valid-but-falsy value like an empty dict or 0 would be wrongly treated the same as None.)

6 Your custom exception is being raised correctly but a caller's error handling is swallowing it. How do you catch that in code review?
Strong answer

Look specifically at what every except block up the call chain catches — except Exception: (or bare except:) will catch a specific custom exception right alongside genuine bugs, and treat both the same way. In review, I'd flag any except Exception: that doesn't re-raise or at least log what it caught, and ask whether the more specific exception type should be caught (and handled meaningfully) instead.

Red flags

Not recognizing that "the exception was raised correctly" and "the exception was handled correctly" are two separate things that both need checking.

Follow-up

"What's the danger of a caught exception that fails silently (no log, no re-raise)?" (There's no signal anywhere that anything went wrong — the only evidence is the operation quietly not doing what it should have.)

7 A recursive function crashes with RecursionError. Walk me through your diagnosis.
Strong answer

The base case either doesn't exist, or exists but can be stepped over rather than landed on exactly. I'd check the base-case condition against every value the recursive parameter could actually take — an equality check like n != 0 is a common trap, since a value that skips past 0 (e.g. counting down from a negative start) never satisfies it. The fix is usually switching to an inequality that can't be skipped, like n <= 0, or validating the input before recursion begins.

Red flags

Suggesting sys.setrecursionlimit() as "the fix" — that just delays the same crash, it doesn't address a genuinely broken or unreachable base case.

Follow-up

"When, if ever, is raising the recursion limit actually the right call?" (Only when the recursion is genuinely correct and bounded, but legitimately deep — e.g., a known, finite, deep tree structure — never as a workaround for a broken base case.)

8 Where should you validate a value that might be zero before it's used as a denominator — right before the division, or earlier?
Strong answer

As close to the division as the function boundary allows, ideally right inside the function that performs the division, since that function is the one that actually knows zero is unsafe there — relying on every caller to remember to check first is fragile and easy to forget once. The check should also decide what "zero reviews" or "zero total" actually means for the caller (a real 0, a None, a "not enough data" message), not just avoid the crash.

Red flags

"Wrap it in try/except ZeroDivisionError" as the only answer — that avoids the crash but doesn't decide what a meaningful zero-case result actually is.

Follow-up

"Is 0 reviews a bug, or a legitimate state?" (Legitimate — a brand-new product with no reviews is normal, so the function needs to handle it as expected input, not as an error case.)

9 What's the difference between FileNotFoundError and PermissionError, and why does conflating them matter?
Strong answer

FileNotFoundError means the path doesn't exist; PermissionError means the path exists but the OS denies the process access to it. They call for genuinely different responses: a missing file is often fine to treat as "nothing here yet," while a permission error usually means the deployment or environment itself is misconfigured and should fail loudly with an actionable message. Code that only catches FileNotFoundError will let a real PermissionError propagate as an unhandled crash in exactly the environment (a locked-down production server) where it's most likely to happen.

Red flags

Treating "file-related exception" as one bucket and reaching for a single broad except OSError: without distinguishing what each specific case should actually do.

Follow-up

"What's a real production scenario where this distinction matters?" (A deploy script that fails on a locked-down directory it doesn't have read access to — treating that identically to "first run, no file yet" hides a genuine infrastructure misconfiguration.)

10 An API call succeeds (200 OK... or does it?) but json.loads() on the response body raises JSONDecodeError. What do you check first?
Strong answer

Whether the code checked the response status code before assuming the body was JSON at all. A very common real-world cause is that the request didn't actually succeed — a login redirect, a rate-limit response, or a "503 Service Unavailable" page, all of which return HTML or plain text, not JSON — and the calling code skipped that check and went straight to json.loads(). I'd check status_code (or equivalent) first, and wrap the parse itself in its own try/except so a genuinely malformed response produces a clear, specific error instead of a raw parser traceback.

Red flags

Assuming "the API sent bad JSON" as the only possible cause, without considering that it may not have sent JSON at all.

Follow-up

"Where's the right layer to handle this — inside the function that calls json.loads(), or the caller of that function?" (Inside the function making the request, right after getting the response — that's the only place that actually knows whether the response looks like an error page before parsing is even attempted.)
