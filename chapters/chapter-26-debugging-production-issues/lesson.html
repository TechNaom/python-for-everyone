#1

KeyError: 'DATABASE_URL'

Symptom

The service crashes on startup, before it ever handles a single request, with a KeyError pointing at a config key.

Root cause

Chapter 8 covered that some_dict[key] raises KeyError the instant the key isn't present — there's no "closest match" or silent fallback. A hardcoded config dict is missing a key that a deployment step (an environment variable, a secrets file) was supposed to provide.

Fix

Use config.get("DATABASE_URL") with an explicit fallback or a clear startup-time error message, instead of letting a raw KeyError (which doesn't say why the key is missing) surface as the crash.

config = {"HOST": "localhost", "PORT": 5432}

def connect():

    return f"Connecting to {db_url}"

#2

IndexError: list index out of range

Symptom

A report-generation function crashes intermittently — it works for most inputs, then suddenly fails with IndexError once a list is shorter than expected.

Root cause

Chapter 5 covered that range(n) doesn't know or care how long any particular list is — this function assumes there are always at least n items to loop over, and that assumption silently holds until the day the list is shorter.

Fix

Bound the loop by the list's own length: range(min(n, len(scores))), or loop over scores[:n] directly so the loop can never ask for an index that isn't there.

scores = [88, 92, 79, 95]

def bonus_for_top_n(scores, n):
    total = 0

    return total

#3

A bare except: silently hides a typo, forever

Symptom

A settings loader "works" — it never crashes, and it always returns a default timeout of 30. Except it's always returning the default, even for configs where a real value was actually provided, and nobody notices for months.

Root cause

Chapter 12 covered that a bare except: catches every exception, not just the one you had in mind. Here the config key is typo'd ("timeoout"), which raises a KeyError — and the bare except: quietly swallows it and returns the fallback, exactly as if the key had simply been left out on purpose. The typo becomes invisible.

Fix

Catch the specific exception you actually expect (except KeyError:), never a bare except: — that way a genuine typo or unrelated bug still surfaces as its own distinct crash instead of being silently absorbed into "using the default."

issue_03_bare_except.py 
def load_settings(raw):
    try:
        return int(raw["timeout"])
    except:
        return 30

# a typo in the config key -- "timeoout", not "timeout"
print(load_settings({"timeoout": "45"}))
actual output (Python 3.14.4) 
30

Nothing here looks broken — the function runs, returns a plausible number, and never raises. That's exactly the danger: change except: to except KeyError: and run it again, and the real bug (a typo'd key that means the configured timeout of 45 is never actually used) finally surfaces as a visible KeyError instead of hiding behind a fallback value forever.

#4

TypeError: unsupported operand type(s) for /: 'str' and 'int'

Symptom

A checkout function crashes with a TypeError several function calls deep — the traceback points at a line of ordinary-looking arithmetic.

Root cause

Chapter 2 covered that Python is dynamically typed — nothing stops a caller from passing a string where a number was expected. Here percent_off arrives as the string "10" (likely straight from form input or a query parameter) and gets passed through two function calls before the arithmetic finally fails, far from where the bad value actually originated.

Fix

Convert and validate external input at the boundary where it enters the system (right after reading it from a form/request), not deep inside business logic — by the time apply_discount() runs, percent_off should already be guaranteed numeric.

def apply_discount(price, percent_off):

def checkout_total(price, percent_off):
    return apply_discount(price, percent_off)

#5

AttributeError: 'NoneType' object has no attribute 'get'

Symptom

Sending a welcome email crashes with an AttributeError mentioning NoneType — a type that was never mentioned anywhere in the code that crashed.

Root cause

Chapter 10 covered that a function with no explicit return on some code path implicitly returns None on that path. find_user() only returns inside the loop when a match is found — if the loop finishes without matching, the function falls through and returns None, with nothing announcing that it happened.

Fix

Check for None right where the value comes back (if user is None: ...) before calling any method on it — or raise a clear, named exception from find_user() itself when no match exists, instead of returning None silently.

users = [

]

def find_user(name):
    for u in users:
        if u["name"] == name:
            return u
    # falls through here if nothing matched -- implicitly returns None

    user = find_user(name)
    return user.get("email").upper()

#6

A custom exception gets raised correctly -- and swallowed anyway

Symptom

A withdrawal of $200 from a $50 balance is reported as successful, with the balance left unchanged at $50 — no error, no warning, and no money actually moved (which is the only reason this bug is survivable at all).

Root cause

Chapter 12 covered writing custom exception classes specifically so callers can react to meaningful, named failures. InsufficientFundsError is raised correctly here — but the caller's except Exception: is broad enough to catch it too, right alongside genuine bugs, and silently falls back to "return the balance unchanged" as if nothing happened.

Fix

Catch the specific custom exception (except InsufficientFundsError:) and handle it deliberately — show the user a real "insufficient funds" message — instead of catching Exception broadly and treating every possible failure the same way.

issue_06_swallowed_custom_exception.py 
class InsufficientFundsError(Exception):
    pass

def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFundsError(f"Cannot withdraw {amount}, balance is {balance}")
    return balance - amount

def process_withdrawal(balance, amount):
    try:
        return withdraw(balance, amount)
    except Exception:
        return balance

result = process_withdrawal(50, 200)
print(f"Balance after withdrawal attempt: {result}")
actual output (Python 3.14.4) 
Balance after withdrawal attempt: 50

The output looks completely reasonable in isolation — it's only wrong once you know a $200 withdrawal was silently rejected instead of clearly refused. This is the same shape of bug as #3: a too-broad except turning a real, specific failure into indistinguishable "normal" behavior.

#7

RecursionError: maximum recursion depth exceeded

Symptom

A recursive countdown function crashes with a huge traceback ending in RecursionError, instead of just counting down and stopping.

Root cause

Chapter 10 covered that every recursive function needs a base case that's actually reachable. countdown()'s base case checks n != 0 — but called with a negative starting number, n counts -5, -6, -7, ... and steps straight past 0 without ever landing exactly on it, so the base case never triggers.

Fix

Use a base case that can't be skipped over, such as n <= 0 instead of n != 0 — or validate the input and reject a negative starting value before recursion begins at all.

def countdown(n):

#9

FileNotFoundError and PermissionError look similar -- and aren't

Symptom

A log reader was written to handle "the file doesn't exist yet" gracefully. In production it crashes anyway — not because the file is missing, but because the process doesn't have permission to read it.

Root cause

Chapter 13 covered handling FileNotFoundError around open(). That's a different, more specific exception than PermissionError (raised when the file exists but the OS denies access — a locked-down production deploy directory, for example). Code that only catches FileNotFoundError lets a real PermissionError propagate as an unhandled crash, and the two are easy to mentally lump together as "file problems" even though the fix for each is completely different.

Fix

Catch both explicitly, and handle them differently: FileNotFoundError usually means "treat this as empty/first run"; PermissionError usually means the deployment itself is misconfigured and should fail loudly with a message an operator can act on, not be silently absorbed.

issue_09_file_errors.py 
def read_log(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        return "no log file yet"

print(read_log("/var/log/myapp/nonexistent-2026-07-09.log"))
actual output (Python 3.14.4) 
no log file yet

This particular run succeeds because the path genuinely doesn't exist. The bug only shows up in an environment where the path exists but is locked down — at that point, open() raises PermissionError instead, which this except FileNotFoundError: does not catch, and the crash propagates uncaught.

#10

json.decoder.JSONDecodeError: Expecting value

Symptom

Parsing an API response crashes with JSONDecodeError — not because the API sent malformed JSON, but because it didn't send JSON at all.

Root cause

Chapter 19 covered json.loads() and checking a response's status code before trusting its body. This code skips that check and assumes every response body is valid JSON — but a real outage, rate limit, or misconfigured URL often returns an HTML error page (a login redirect, a "503 Service Unavailable" page) instead, and json.loads() has no way to parse that as JSON.

Fix

Check response.status_code (or equivalent) before parsing, and wrap json.loads() in its own try/except json.JSONDecodeError so a non-JSON response produces a clear "the API returned something unexpected" error instead of a raw parser traceback.

import json

Quick recap
  • Most production crashes trace back to an assumption that was true in testing and false in the real world: a key that's always present, a list that's always long enough, input that's always the right type, a denominator that's never zero.
  • A bare except: or an overly broad except Exception: doesn't prevent bugs — it hides them, often turning a loud, easy-to-diagnose crash into a silent wrong answer (issues #3 and #6).
  • Similarly-named exceptions (FileNotFoundError vs. PermissionError) can require genuinely different handling — catching one doesn't mean the other is covered.
In the Real World (Category 1)
  • Production incident write-ups almost always follow this exact shape — symptom, root cause, fix — because that's the order information actually becomes available: you see the crash first, then dig into why, then decide what changes.
  • The most dangerous bugs in real systems are rarely the ones that crash loudly (issues #1, #2, #4, #5, #7, #8, #10 here) — they're the ones that silently return a plausible-but-wrong answer (issues #3 and #6), because nothing alerts anyone that something went wrong.
  • Being able to predict which exception a piece of code is about to throw — before running it — is a skill senior engineers use constantly during code review, and it's built entirely from knowing these patterns cold.

#11

Checking membership against a growing list, over and over

Symptom

A "find duplicates" function that's fast on a small test list becomes noticeably slow once the real dataset has a few thousand items — and gets dramatically worse as the dataset grows, not just proportionally worse.

Root cause

Chapter 8 covered that checking item in some_list has to scan the list from the start until it finds a match (or reaches the end) — an O(n) operation. Doing that check inside a loop that runs n times makes the whole function O(n²): every new item re-scans an ever-growing list.

Fix

Use a set instead of a list for the membership check — Chapter 8 also covered that item in some_set is O(1) on average, regardless of how large the set gets.

issue_11_quadratic_membership.py 
import time

def find_duplicates_slow(items):
    seen = []
    duplicates = []
    for item in items:
        if item in seen:  # O(n) list scan, called n times -> O(n^2)
            duplicates.append(item)
        else:
            seen.append(item)
    return duplicates

def find_duplicates_fast(items):
    seen = set()
    duplicates = []
    for item in items:
        if item in seen:  # O(1) average-case set membership
            duplicates.append(item)
        else:
            seen.add(item)
    return duplicates

data = list(range(5000)) + list(range(2000))

start = time.perf_counter()
find_duplicates_slow(data)
slow_time = time.perf_counter() - start

start = time.perf_counter()
find_duplicates_fast(data)
fast_time = time.perf_counter() - start

print(f"list-based (O(n^2)): {slow_time:.4f}s")
print(f"set-based (O(n)):    {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
list-based (O(n^2)): 0.0972s
set-based (O(n)):    0.0008s

Thought process: the actual fix here isn't "sets are faster than lists" as a blanket rule — it's recognizing the specific access pattern (repeated membership checks against a growing collection) where the difference between O(1) and O(n) per check compounds into O(n) vs. O(n²) overall. An experienced dev looks for this exact shape: any in check inside a loop, against a collection that's also being built up inside that same loop.

#12

Inserting at the front of a list in a loop

Symptom

A function that reverses the order of a list by building it up one item at a time gets dramatically slower as the input grows, even though it's doing the same amount of "work" per item.

Root cause

Chapter 7 covered that list.insert(0, x) has to shift every existing element over by one to make room at the front — an O(n) operation. Calling it n times to build an n-item list makes the whole function O(n²), the same shape of problem as issue #11 but from a different list operation.

Fix

list.append() adds to the end in O(1) amortized time. Append everything in order, then call .reverse() once at the very end — one O(n) pass total, instead of n separate O(n) shifts.

issue_12_insert_front.py 
import time

def reverse_order_slow(items):
    result = []
    for item in items:
        result.insert(0, item)  # O(n) shift, every single call
    return result

def reverse_order_fast(items):
    result = []
    for item in items:
        result.append(item)  # O(1) amortized
    result.reverse()  # one O(n) pass at the end
    return result

items = list(range(20000))

start = time.perf_counter()
a = reverse_order_slow(items)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = reverse_order_fast(items)
fast_time = time.perf_counter() - start

print(a == b)
print(f"insert(0, x) in a loop: {slow_time:.4f}s")
print(f"append then reverse():  {fast_time:.4f}s")

#13

Recomputing the same expensive result over and over

Symptom

A checkout function that calculates tax for a shopping cart is slow specifically when the cart has many items from the same handful of regions — even though there's only a few distinct regions actually involved.

Root cause

Chapter 10 covered function calls as the unit of reusable work — but nothing about a function call automatically remembers a previous result. A tax-rate lookup was called once per cart item, even though the same region's rate was computed identically, from scratch, every single time it recurred.

Fix

Cache results in a dict (Chapter 8) keyed by the input that determines the result — here, the region. Look up the cache before doing the expensive work, and only do the work (and store it) the first time a given region is seen.

issue_13_repeated_recompute.py 
import time

def slow_tax_lookup(region):
    # stands in for a genuinely expensive computation
    total = 0
    for _ in range(20000):
        total += 1
    return 0.08

def total_price_slow(cart, get_tax_rate):
    total = 0
    for item in cart:
        rate = get_tax_rate(item["region"])  # recomputed for every item
        total += item["price"] * (1 + rate)
    return total

def total_price_fast(cart, get_tax_rate):
    total = 0
    cache = {}
    for item in cart:
        region = item["region"]
        if region not in cache:
            cache[region] = get_tax_rate(region)
        total += item["price"] * (1 + cache[region])
    return total

cart = [{"price": 10, "region": "CA"} for _ in range(500)]

start = time.perf_counter()
total_price_slow(cart, slow_tax_lookup)
slow_time = time.perf_counter() - start

start = time.perf_counter()
total_price_fast(cart, slow_tax_lookup)
fast_time = time.perf_counter() - start

print(f"recompute every item: {slow_time:.4f}s")
print(f"cache by region:      {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
recompute every item: 0.3718s
cache by region:      0.0007s

Thought process: caching is only a correct fix when the same input reliably produces the same output — the region's tax rate doesn't change mid-checkout, so caching by region is safe here. An experienced dev checks that assumption explicitly before reaching for a cache, since caching a result that legitimately changes between calls (a live stock price, a random value) would introduce a correctness bug, not just a performance win.

#14

A nested loop to join two lists of records

Symptom

A function that attaches customer names to a list of orders works fine in testing with a handful of records, then times out once it's run against the real orders and customers tables.

Root cause

For every order, the code scans the entire customers list looking for a matching ID — an O(m) scan repeated n times, for O(n·m) overall. With a few thousand of each, that's tens of millions of comparisons for what should be a simple lookup.

Fix

Build a dict mapping customer ID to customer record once, up front (Chapter 8's dict/set-comprehension-friendly pattern), then look up each order's customer in O(1) instead of scanning — O(n + m) overall instead of O(n·m).

issue_14_nested_loop_lookup.py 
import time

def enrich_orders_slow(orders, customers):
    enriched = []
    for order in orders:
        for customer in customers:  # full scan, for EVERY order
            if customer["id"] == order["customer_id"]:
                enriched.append({**order, "customer_name": customer["name"]})
                break
    return enriched

def enrich_orders_fast(orders, customers):
    customer_by_id = {c["id"]: c for c in customers}  # built once
    enriched = []
    for order in orders:
        customer = customer_by_id.get(order["customer_id"])
        enriched.append({**order, "customer_name": customer["name"] if customer else None})
    return enriched

customers = [{"id": i, "name": f"Customer {i}"} for i in range(2000)]
orders = [{"customer_id": i % 2000, "amount": 10} for i in range(2000)]

start = time.perf_counter()
a = enrich_orders_slow(orders, customers)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = enrich_orders_fast(orders, customers)
fast_time = time.perf_counter() - start

print(a == b)
print(f"nested loop (O(n*m)): {slow_time:.4f}s")
print(f"dict lookup (O(n)):   {fast_time:.4f}s")

Recompiling the same regular expression on every call

Symptom

A log-scanning function using re.search() is a bit slower than expected on a large batch of lines — a real but modest overhead, not a dramatic one.

Root cause

Chapter 18 covered re.search(pattern, text) as a convenience function — internally, it still has to turn the pattern string into a compiled matcher before it can search. Called repeatedly with the same pattern string, that parsing step repeats every time (CPython does cache a small number of recently-used patterns internally, which is why this speedup is real but modest rather than dramatic).

Fix

Call re.compile(pattern) once, outside the loop, and call .search()/.match() on the resulting compiled pattern object instead — the parsing step happens exactly once no matter how many times it's used afterward.

import re
import time

    found = []
    for line in lines:
        match = re.search(PATTERN_STR, line)  # parses PATTERN_STR every call
        if match:
            found.append(match.group())
    return found

def find_phone_numbers_fast(lines):

    found = []
    for line in lines:
        match = compiled.search(line)
        if match:

    return found

fast_time, b = min((timed(find_phone_numbers_fast, lines) for _ in range(5)), key=lambda t: t[0])

#17

Symptom

A hand-written while loop summing a very large list is measurably slower than the equivalent for loop, even though both do the same O(n) amount of "real" work.

Root cause

This one isn't about Big-O — len() on a list is already O(1). It's about repeated small overhead: while i < len(items): calls len() fresh on every single iteration, and that function-call overhead adds up across millions of iterations even though each individual call is cheap.

Fix

Compute len(items) once, store it in a variable before the loop starts, and check against that variable instead — the list's length doesn't change during the loop here, so there's nothing to recompute.

#19

Materializing a full list when the loop could stop early

Symptom

A "find the first value over a threshold" function is dramatically slower than it should be on a large dataset, even though it usually only needs to look at the first handful of values before finding a match.

Root cause

Chapter 17 covered generators as lazy, on-demand value production. Fix

#21

An unbounded module-level cache that never shrinks

Symptom

A service's memory usage climbs steadily over days of uptime and never comes back down, even during quiet periods — eventually the process is killed by the host for using too much memory.

Root cause

#22

A reference cycle that survives until the garbage collector runs

Symptom

Objects that should be destroyed the moment they go out of scope stick around instead — a tree of parent/child objects doesn't get cleaned up immediately after the code that built it returns.

Root cause

Chapter 25 covered that CPython primarily uses reference counting: an object is freed the instant its reference count hits zero. A parent that holds a reference to its child, while the child holds a reference back to its parent, creates a cycle — each object's count never reaches zero on its own, even after nothing outside the cycle refers to either one. Only the separate, periodic cyclic garbage collector (gc) can break cycles like this, and it doesn't run instantly.

Fix

Break the cycle deliberately — store the child-to-parent link as a weakref.ref instead of a normal reference (issue #30 below takes this further), so the parent can still be freed by simple reference counting the moment nothing outside the cycle needs it.

issue_22_circular_del.py 
import gc

class Node:
    def __init__(self, name):
        self.name = name
        self.parent = None
        self.children = []

    def __del__(self):
        print(f"Node {self.name} destroyed")

def build_tree():
    root = Node("root")
    child = Node("child")
    root.children.append(child)
    child.parent = root  # circular reference: root -> child -> root
    return root

gc.disable()
build_tree()
print("build_tree() returned, but no destruction message printed yet")
gc.enable()
gc.collect()
print("after gc.collect(), the cycle is finally cleaned up")
actual output (Python 3.14.4)

Node child destroyed
after gc.collect(), the cycle is finally cleaned up

With gc enabled (the default), CPython's cyclic collector runs periodically on its own and would eventually catch this — the danger in a real long-running service is the gap between "logically unreachable" and "actually collected," during which every leaked cycle's memory stays held.

#23

Symptom

Root cause

Chapter 10 covered passing functions (including bound methods) as values. #30

A registry that keeps objects alive by design, unintentionally

Symptom

A session registry meant only to look up active sessions ends up being the reason sessions never get garbage collected, even long after every other part of the program has finished with them.

Root cause

A plain dict holds a strong reference to every value it contains — that's normal dict behavior, and usually exactly what's wanted. But a registry whose entire job is "let me look this up if it still exists" doesn't need to be the thing keeping it alive; here, every Session the caller creates and discards stays alive anyway, purely because the registry still references it.

Fix

Use weakref.WeakValueDictionary instead of a plain dict when a collection's purpose is tracking, not owning — entries disappear automatically the moment nothing else references the value, which is usually the actual intent of a "registry."

issue_30_weakref_fix_demo.py 
import weakref

class Session:
    def __init__(self, session_id):
        self.session_id = session_id

class SessionRegistry:
    def __init__(self):
        self._sessions = {}  # plain dict -- holds a strong reference forever

    def register(self, session):
        self._sessions[session.session_id] = session

    def count(self):
        return len(self._sessions)

class SessionRegistryFixed:
    def __init__(self):
        self._sessions = weakref.WeakValueDictionary()  # doesn't keep sessions alive

    def register(self, session):
        self._sessions[session.session_id] = session

    def count(self):
        return len(self._sessions)

registry = SessionRegistry()
for i in range(1000):
    s = Session(i)
    registry.register(s)
    # caller drops its reference to s here -- but the registry keeps it alive anyway
print(f"leaky registry still holding: {registry.count()} sessions")

registry_fixed = SessionRegistryFixed()
for i in range(1000):
    s = Session(i)
    registry_fixed.register(s)
    # caller drops its reference; WeakValueDictionary lets it actually be collected
print(f"weakref-based registry holding: {registry_fixed.count()} sessions")
actual output (Python 3.14.4) 
leaky registry still holding: 1000 sessions
weakref-based registry holding: 1 sessions

The weakref-based registry ends up holding just 1 session — the very last one, still referenced by the loop variable s at the moment count() runs. Every earlier session was already collected the instant its last strong reference went away.

Quick recap
  • A memory leak in Python almost always means "something is still reachable that shouldn't be" — not a language failure, but an object graph that keeps a path alive longer than intended.
  • Caches, registries, closures, and class attributes are the most common places an unbounded collection hides, because each one looks like ordinary, correct code in isolation.
  • weakref (issues #23 and #30) and bounded collections like collections.deque(maxlen=...) or lru_cache(maxsize=...) (issues #21, #25, #29) are the two main tools for turning "grows forever" into "grows up to a limit."
In the Real World (Category 3)
  • Memory leaks are notoriously hard to catch in testing because tests are short-lived — a leak that adds a few kilobytes per request is invisible in a test suite and catastrophic after two weeks of production uptime.
  • "Restart the service every night" is a real, common mitigation teams use while the actual leak gets tracked down — it's not a fix, but it buys time without an outage.
  • Tools like tracemalloc (used in issue #28) and external profilers exist specifically because "where is the memory going" is much harder to answer by reading code than "why did this line crash" is — leaks are diagnosed by measurement, not inspection.

#31

A race condition on a shared counter with no lock

Symptom

Eight threads each increment a shared counter by a known amount, but the final total comes out lower than the sum of what every thread should have contributed — and the exact shortfall changes from run to run.

Root cause

Chapter 20 covered starting and joining threads, but counter += 1 is not one atomic step — it's read counter, add 1, write the result back. If two threads both read the same value before either writes its update back, one thread's increment is silently lost. Python's GIL (Global Interpreter Lock) prevents two threads from running Python bytecode at the exact same instant, but it does not prevent a thread from being switched out in the middle of a multi-step operation like this one — which is exactly where the race lives.

Fix

Protect the read-modify-write with a threading.Lock (issue #40 below) so only one thread can execute that sequence at a time — turning three separate steps back into one effectively atomic one.

issue_31_race_condition.py 
import threading
import time

counter = 0

def increment(times):
    global counter
    for _ in range(times):
        current = counter
        time.sleep(0)  # force a thread-switch opportunity between read and write
        counter = current + 1

threads = [threading.Thread(target=increment, args=(2000,)) for _ in range(8)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(f"expected: {8 * 2000}")
print(f"actual:   {counter}")
actual output (Python 3.14.4) -- the exact number lost to the race varies by run; this is a single real run, not a fixed number 
expected: 16000
actual:   2002

The time.sleep(0) here exists purely to make this race reliably visible on the page — it forces Python to check for a thread switch right between the read and the write, widening a gap that's normally too narrow to hit often. Without it, this exact code would still be broken, just less predictably so — which is part of what makes races so dangerous: the bug is real even on runs where nothing looks wrong.

Thought process: "it worked when I tested it" is close to meaningless for concurrency bugs — a race condition can pass every test run and still be present, because whether it manifests depends on exact timing that testing doesn't control. The fix has to be reasoned about structurally (is this shared state ever mutated by more than one thread without a lock?), not verified by running the code a few times and seeing no error.

#32

Deadlock from acquiring two locks in different orders

Symptom

Two threads that each transfer funds between two accounts — one A-to-B, one B-to-A — occasionally hang forever, with both threads stuck and neither making progress.

Root cause

transfer_a_to_b() acquires lock_a then lock_b; transfer_b_to_a() acquires lock_b then lock_a — the opposite order. If both threads grab their first lock at roughly the same time, each ends up waiting forever for the lock the other thread is holding. Neither thread can proceed, and neither will ever release what it's holding, since the release happens only after the wait — a classic deadlock.

Fix

Establish one global ordering for acquiring locks (e.g. always sort accounts by ID and lock the lower one first) and have every code path follow it — if both functions always acquired lock_a before lock_b, the deadlock in this exact shape becomes structurally impossible.

issue_32_deadlock_lock_order.py 
import os
import sys
import threading
import time

lock_a = threading.Lock()
lock_b = threading.Lock()

def transfer_a_to_b():
    with lock_a:
        time.sleep(0.1)  # simulate some work, widening the window for the deadlock
        with lock_b:
            print("transferred A -> B")

def transfer_b_to_a():
    with lock_b:
        time.sleep(0.1)
        with lock_a:
            print("transferred B -> A")

t1 = threading.Thread(target=transfer_a_to_b, daemon=True)
t2 = threading.Thread(target=transfer_b_to_a, daemon=True)
t1.start()
t2.start()
t1.join(timeout=3)
t2.join(timeout=3)

print(f"t1 still alive (deadlocked): {t1.is_alive()}")
print(f"t2 still alive (deadlocked): {t2.is_alive()}")
sys.stdout.flush()
os._exit(1)  # both daemon threads are still holding a lock forever -- force process exit
actual output (Python 3.14.4) 
t1 still alive (deadlocked): True
t2 still alive (deadlocked): True

daemon=True and the manual os._exit(1) at the end exist only so this demo terminates instead of hanging the page's build process forever — both threads really are stuck waiting on each other permanently. A real deadlocked process, without that escape hatch, would simply hang until something external (an operator, a health-check timeout) killed it.

#33

Threading a CPU-bound task and expecting a speedup

Symptom

Splitting a pure-computation task across two threads, expecting roughly a 2x speedup on a multi-core machine, produces no speedup at all — sometimes it's even slightly slower.

Root cause

Chapter 20 covered that CPython's GIL allows only one thread to execute Python bytecode at a time, no matter how many CPU cores are available. Threading genuinely helps when threads spend most of their time waiting (for a network response, for disk I/O) — the GIL is released during that wait. A tight pure-Python loop like count_down() never waits for anything; it's 100% CPU-bound Python bytecode, so the GIL keeps the two threads running one at a time, exactly as if they'd run sequentially (plus a bit of thread-switching overhead).

Fix

For CPU-bound work, use multiprocessing instead of threading — separate processes each get their own Python interpreter and their own GIL, so they genuinely run in parallel across cores. Reserve threading for I/O-bound work, where waiting (not computing) is most of what a thread does.

issue_33_gil_cpu_bound.py 
import threading
import time

def count_down(n):
    while n > 0:
        n -= 1

N = 30_000_000

start = time.perf_counter()
count_down(N)
count_down(N)
single_threaded_time = time.perf_counter() - start

start = time.perf_counter()
t1 = threading.Thread(target=count_down, args=(N,))
t2 = threading.Thread(target=count_down, args=(N,))
t1.start(); t2.start()
t1.join(); t2.join()
two_threads_time = time.perf_counter() - start

print(f"single-threaded, sequential: {single_threaded_time:.3f}s")
print(f"two threads, 'parallel':     {two_threads_time:.3f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
single-threaded, sequential: 2.164s
two threads, 'parallel':     2.182s

No speedup — if anything, a hair slower, purely from thread-management overhead. This is the single most common GIL misconception in practice: "more threads" isn't "more parallelism" for CPU-bound Python code, no matter how many cores the machine has.

#34

Forgetting to join a thread before reading its result

Symptom

A function that kicks off background work in threads and then immediately checks the results reports an empty list, even though the work clearly did finish (just a fraction of a second later).

Root cause

Chapter 20 covered thread.start() as non-blocking: it schedules the thread to run and returns immediately, without waiting for it to finish. run_batch_missing_join() starts three threads but never calls t.join() on any of them — so it returns before any of the threads have necessarily completed their time.sleep(0.3) plus computation, and results is still empty at that moment.

Fix

Call t.join() on every thread the function started before reading anything those threads were supposed to produce — join() blocks until that specific thread has actually finished, which is the whole point of it.

#35

A producer/consumer queue with no way to signal "done"

Symptom

A background consumer thread reads from a queue and processes everything correctly — and then the whole program hangs, never exiting, even though there's clearly nothing left to do.

Root cause

Chapter 20's threading model extends naturally to queue.Queue for producer/consumer patterns — but q.get() blocks by default when the queue is empty, waiting for the next item to arrive. The producer here puts three items and then simply stops, with no mechanism telling the consumer "that was the last one." The consumer's while True: q.get() is left waiting on a fourth item that will never come.

Fix

Have the producer put a sentinel value (like None) onto the queue once it's done, and have the consumer break its loop when it sees that sentinel — or use q.join()/task_done() together with a separate shutdown signal so the consumer knows precisely when to stop waiting.

issue_35_queue_producer_consumer.py 
import queue
import threading

def producer(q, items):
    for item in items:
        q.put(item)
    # BUG: never puts a sentinel value or otherwise signals "no more items"

def consumer(q, results):
    while True:
        item = q.get()  # blocks forever once the queue is empty and never told to stop
        results.append(item * 2)
        q.task_done()

q = queue.Queue()
results = []

t_producer = threading.Thread(target=producer, args=(q, [1, 2, 3]), daemon=True)
t_consumer = threading.Thread(target=consumer, args=(q, results), daemon=True)
t_producer.start()
t_consumer.start()
t_producer.join(timeout=2)

def wait_for_queue():
    q.join()
    q_drained.set()

waiter = threading.Thread(target=wait_for_queue, daemon=True)
waiter.start()
waiter.join(timeout=1)

print(f"producer finished: {not t_producer.is_alive()}")

#36

Check-then-act on a shared list from multiple threads

Symptom

A pool of worker threads pulling tasks off a shared list occasionally crashes with IndexError: pop from empty list, even though every worker explicitly checks while shared_queue: before popping.

Root cause

The check (if not shared_queue) and the act (shared_queue.pop()) are two separate steps, and nothing stops another thread from running its own pop in between them. Thread A can see a non-empty list, get interrupted, and by the time it actually calls .pop(), thread B has already emptied it — the check was true when made, but false by the time it's acted on.

Fix

Make the check-and-pop a single atomic operation instead of two separate steps — wrap both in the same with lock: block, or catch IndexError around the pop() and treat it as "someone else got there first" rather than a fatal error.

issue_36_thread_unsafe_list_mutation.py 
import sys
import threading
import time

sys.setswitchinterval(0.0000001)

shared_queue = [f"task-{i}" for i in range(200)]

def worker():
    while True:
        if not shared_queue:  # check
            return
        time.sleep(0)  # force a thread-switch opportunity between check and act
        try:
            task = shared_queue.pop()  # act -- another thread may have emptied it by now
        except IndexError as e:
            errors.append(e)
            return
        processed.append(task)

    t.start()
for t in threads:
    t.join()

sys.setswitchinterval() and the time.sleep(0) are there purely to widen the timing window so the race reliably shows up in a short script — the underlying bug (check-then-act on shared mutable state) is exactly as real without them, just harder to trigger on demand.

#37

A plain Lock deadlocking against itself

Symptom

A function that acquires a lock, then calls a second function that also tries to acquire the very same lock, hangs forever — on the very first call, with no other threads involved at all.

Root cause

A plain threading.Lock doesn't track which thread holds it — once acquired, any further attempt to acquire it blocks, even from the exact same thread that's already holding it. outer() acquires lock, then calls inner(), which tries to acquire the same lock again from within the same thread — and blocks waiting for itself to release something it's currently holding.

Fix

Use threading.RLock (a re-entrant lock) when the same thread legitimately needs to acquire the same lock more than once, such as one locked method calling another — an RLock tracks the owning thread and lets it re-acquire without blocking on itself.

issue_37_lock_not_reentrant.py 
import threading

lock = threading.Lock()

def outer():
    with lock:
        inner()  # tries to acquire the SAME plain Lock again -- deadlocks immediately

def inner():
    with lock:
        print("inner acquired the lock")

t = threading.Thread(target=outer, daemon=True)
t.start()
t.join(timeout=2)
print(f"thread still blocked (self-deadlocked): {t.is_alive()}")
actual output (Python 3.14.4) 
thread still blocked (self-deadlocked): True

Switching lock = threading.Lock() to lock = threading.RLock() is the entire fix — with that one change, outer() prints "inner acquired the lock" and the thread finishes normally instead of hanging.

#40

Fixing issue #31 with a lock

Symptom

N/A — this entry shows the fix for issue #31's race condition, run the same way, to confirm the lock actually closes the gap rather than just making the race less frequent.

Root cause

Issue #31's counter += 1 was unsafe specifically because it's three separate steps that another thread can interleave with. Wrapping those same three steps in with lock: means only one thread can be inside that block at any moment — any other thread calling increment_safe() simply waits at the with lock: line until the current holder finishes and releases it.

    global counter
    for _ in range(times):
        with lock:  # only one thread at a time can read-modify-write counter
            current = counter
            current = current + 1
            counter = current

threads = [threading.Thread(target=increment_safe, args=(2000,)) for _ in range(8)]
for t in threads:
    t.start()
for t in threads:
    t.join()

expected: 16000
actual:   16000

Unlike issue #31, this version produces exactly 16000 every single time it's run, no matter how many times — that consistency, not just one lucky correct run, is what actually proves a concurrency fix works.

Quick recap
  • The GIL prevents two threads from running Python bytecode simultaneously, but it does not make multi-step operations (x += 1, check-then-act, iterate-while-mutating) atomic — races are still very real in pure Python code.
  • Deadlocks (issues #32, #37) come from lock-acquisition order or re-entrancy, not from randomness — they're deterministic bugs that happen to depend on timing to trigger.
  • Threading speeds up I/O-bound work (waiting on a network, a disk, a queue) but not CPU-bound work (issue #33) — for CPU-bound parallelism, multiprocessing is the right tool, not more threads.
In the Real World (Category 4)
  • Concurrency bugs are notorious for passing code review and every test run, then showing up for the first time under real production load — timing-dependent bugs need real concurrent load to reliably surface, which most test suites don't generate.
  • "It only happens sometimes" is the single most common description of a race condition or deadlock in an incident report — that inconsistency is itself a strong diagnostic signal, not a reason to dismiss the report.
  • Production incident postmortems for deadlocks almost always end with "we established a consistent lock ordering" (issue #32) as the fix — it's one of the few concurrency bugs with a genuinely mechanical, reliable solution once identified.

#41

The mutable default argument trap

Symptom

A "start a new shopping cart" function occasionally hands a customer a cart that already has someone else's items in it, with no code anywhere that looks like it should share carts between customers.

Root cause

Chapter 10 covered default argument values — what it's easy to miss is that a default value is evaluated exactly once, when the def statement runs, not fresh on every call. cart=[] creates one single list object at function-definition time; every call that doesn't pass its own cart argument gets that same object, and any mutation (like .append()) persists onto the next such call.

Fix

Use cart=None as the default, and create a fresh list inside the function body when cart is None — def add_item(item, cart=None): cart = [] if cart is None else cart. This is one of the most well-known Python gotchas precisely because the code reads as obviously correct until you know to look for it.

def add_item(item, cart=[]):  # BUG: default list is created ONCE, at def time, and reused
    cart.append(item)
    return cart

print(f"cart_a: {cart_a}")

cart_b = add_item("banana")  # a brand-new customer's cart -- should start empty
print(f"cart_b: {cart_b}")

print(f"same list object: {cart_a is cart_b}")
actual output (Python 3.14.4) 
cart_a: ['apple']
cart_b: ['apple', 'banana']
same list object: True

#42

Shallow copy vs. deep copy

Symptom

Making a copy of an order to modify independently ("just change the status, don't touch the original") ends up silently modifying the original order too — but only its list-valued fields, not its plain string fields.

Root cause

Chapter 8 and Chapter 15 covered dict.copy() as a shallow copy: it creates a new dict, but each value inside it is the same object as in the original, not a new copy of that value. Reassigning a key (order_a["status"] = "shipped") only affects the new dict, because that replaces which object the key points to. But mutating a shared nested object in place (order_a["items"].append(...)) changes the one list both dicts are still pointing at.

Fix

Use copy.deepcopy() when a copy needs to be fully independent, including everything nested inside it — it recursively copies every mutable value, not just the top-level container, so mutating the copy never touches the original.

issue_42_shallow_copy.py 
import copy

order_template = {"items": ["notebook"], "status": "pending"}

order_a = order_template.copy()  # BUG: shallow copy -- "items" list is still shared
order_a["status"] = "shipped"      # this is fine, strings are immutable
order_a["items"].append("pen")     # this mutates the SAME list as order_template's

print(f"order_template: {order_template}")
print(f"order_a:        {order_a}")

order_b = copy.deepcopy(order_template)
order_b["items"].append("eraser")
print(f"order_template after deepcopy mutation: {order_template}")
print(f"order_b:                                 {order_b}")
actual output (Python 3.14.4) 
order_template: {'items': ['notebook', 'pen'], 'status': 'pending'}
order_a:        {'items': ['notebook', 'pen'], 'status': 'shipped'}
order_template after deepcopy mutation: {'items': ['notebook', 'pen'], 'status': 'pending'}
order_b:                                 {'items': ['notebook', 'pen', 'eraser'], 'status': 'pending'}

Note that order_template's items already shows 'pen' from the shallow-copy mutation earlier — the deepcopy in the second half correctly stays independent from that point forward, adding 'eraser' only to order_b.

#43

Comparing floating-point numbers with ==

Symptom

An account reconciliation check flags a balance as "not fully funded" even though, by every human reading of the numbers involved, it clearly should be exactly $1.00.

Root cause

Chapter 2 introduced floats; what it's easy to gloss over is that most decimal fractions (including something as simple as 0.1) can't be represented exactly in binary floating point — the same way 1/3 can't be written exactly in decimal. Each addition accumulates a tiny rounding error, and after ten additions, the result is off from 1.0 by a sliver too small to see when printed casually, but large enough that == reports False.

Fix

Never compare floats for exact equality when they result from arithmetic — use math.isclose(a, b) for a tolerance-based comparison, or use decimal.Decimal instead of float entirely for money, where exact decimal representation actually matters.

account_balance = 0.0

print(f"is it exactly equal to 1.0? {account_balance == 1.0}")

    print("account fully funded")
else:

#44

Silent truncation when coercing types

Symptom

An inventory system that's supposed to track whole units ends up losing small fractional amounts on every single item that arrives with a fractional quantity — no error is ever raised.

Root cause

Chapter 2 covered int() as a way to convert to an integer — what it didn't stress is that int() truncates toward zero rather than rounding, and does so completely silently. int(float("4.9")) doesn't round to 5; it discards everything after the decimal point and returns 4, with nothing in the output to suggest data was thrown away.

Fix

Decide deliberately what fractional quantities should mean before converting — round() if the nearest whole number is correct, an explicit validation error if fractional units should never be allowed at all, or keep the value as a float/Decimal if fractions are actually meaningful data, rather than letting int() make that decision implicitly.

issue_44_silent_truncation.py 
def parse_quantity(raw_value):
    return int(float(raw_value))  # BUG: truncates toward zero instead of rounding or rejecting

# upstream calculation) supplied a fractional value

quantities = [parse_quantity(v) for v in raw_inputs]

print(f"parsed as:   {quantities}")

actual output (Python 3.14.4)

#46

An off-by-one error in pagination that silently drops records

Symptom

A paginated export of 25 records, fetched page by page, comes back missing the first 10 records entirely — with no error, and every individual page request "succeeding."

Root cause

Chapter 6 and Chapter 7 covered slicing as zero-based — records[0:10] is the first ten items. This API's page_number is 1-based (page 1 is the first page, matching how a human thinks about it), but the slicing math treats it as if it were already 0-based: start = page_number * page_size computes 1 * 10 = 10 for "page 1," which is actually where the second page of results starts.

Fix

Convert the 1-based page number to a 0-based offset explicitly before slicing: start = (page_number - 1) * page_size. Writing a quick unit test for "page 1 includes the very first record" would have caught this immediately — it's exactly the kind of boundary case Chapter 12-adjacent testing habits are meant to cover.

records = list(range(1, 26))  # 25 records total, IDs 1-25

#47

Assuming dict key order matches a specific column order

Symptom

A CSV export built by joining a dict's .values() together produces rows where the columns are in a different order depending on which code path constructed each record — even though every record has the exact same fields.

Root cause

Chapter 8 covered that modern Python dicts preserve insertion order — a real, documented guarantee since Python 3.7. The bug here isn't that the guarantee is false; it's relying on it for something it was never meant to guarantee: which order a dict's keys end up in depends entirely on the order they were first assigned, which can differ between two code paths that build "the same" record from different sources.

Fix

Never rely on a dict's insertion order to imply a fixed column layout — explicitly list the field order once (e.g. a tuple of column names) and build every row by looking up each field from that fixed list: [record[field] for field in COLUMN_ORDER], so all rows are aligned regardless of how each record was originally built.

#48

Two visually identical strings that aren't equal

Symptom

A user's account lookup by name fails intermittently — the name displayed on screen looks exactly right, character for character, but == against the stored value returns False.

Root cause

Chapter 6 covered strings as sequences of characters, but some accented characters can be represented two different ways in Unicode: as one single "precomposed" code point, or as a plain letter followed by a separate combining accent mark. Both render identically on screen, but they are different sequences of code points — different lengths, even — so Python's == correctly reports them as unequal strings.

Fix

Normalize any text that crosses a system boundary (typed by a user, read from a file, received from an API) to one consistent Unicode form — typically NFC — using unicodedata.normalize("NFC", text), before comparing it or storing it as a lookup key.

issue_48_unicode_normalization.py 
import unicodedata

# two different byte sequences that render identically on screen --

name_precomposed = "Café"

print(f"precomposed form: {name_precomposed!r} ({len(name_precomposed)} chars)")
print(f"do they print the same? {name_decomposed} == {name_precomposed}")
print(f"are they equal as strings? {name_decomposed == name_precomposed}")

print(f"lookup with the decomposed form found the account? {name_decomposed in registered_users}")

normalized_a = unicodedata.normalize("NFC", name_decomposed)
normalized_b = unicodedata.normalize("NFC", name_precomposed)
print(f"after NFC normalization, equal? {normalized_a == normalized_b}")

#49

Silent wraparound when packing a value into a fixed-width field

Symptom

A daily event counter, serialized into a fixed-width field for a wire protocol, reports a wildly wrong (and much smaller) number on unusually busy days, while working perfectly on normal days.

Root cause

Chapter 3 covered bitwise operators including &. Masking a value with & 0xFFFFFFFF keeps only its lowest 32 bits — for any value that already fits in 32 bits, this is a no-op. But for a value that exceeds 2³² - 1 (4,294,967,295), the higher bits are silently discarded, producing a completely different, smaller number with no error or warning of any kind — unlike issue #4's TypeError, this is a valid Python int operation the whole way through.

Fix

Validate the value's range before masking or packing it into a fixed-width field, and raise an explicit error (or widen the field) when it doesn't fit — Python's own integers have no size limit, so this kind of overflow only happens at a boundary you construct yourself, and only that boundary can catch it.

#50

Truncating by character count when the limit is actually in bytes

Symptom

A username that fits comfortably under a database column's stated character limit still fails to insert, with the database complaining the value is too long for the column.

Root cause

Chapter 6 covered len() on a string as the number of characters. Many database column types (a fixed-width VARBINARY, or VARCHAR under certain encodings) actually enforce a limit in bytes, not characters. Once text is encoded to UTF-8, characters outside the basic ASCII range take up 2, 3, or 4 bytes each — so a 10-character string can easily be 12+ bytes, silently exceeding a byte-based limit that a character-based truncation never accounted for.

Fix

Know which unit a given limit is actually measured in, and truncate accordingly — for a byte limit, encode first, truncate the bytes (carefully, so as not to split a multi-byte character in half), then decode back, rather than truncating the original string by character count and hoping it happens to fit.

issue_50_str_bytes_boundary.py 
def truncate_username(name, max_bytes):

    # a BYTE limit (e.g. a fixed-width database column) -- multi-byte
    # characters make this silently wrong
    return name[:max_bytes]

username = "Zoë_Müller99"  # contains multi-byte UTF-8 characters (ë, ü)
truncated = truncate_username(username, 10)
truncated_bytes = truncated.encode("utf-8")

print(f"original: {username!r} ({len(username)} chars, {len(username.encode('utf-8'))} bytes)")
print(f"'truncated to 10': {truncated!r} ({len(truncated)} chars, {len(truncated_bytes)} bytes)")
print(f"fits in a 10-byte column? {len(truncated_bytes) <= 10}")
actual output (Python 3.14.4) 
original: 'Zoë_Müller99' (12 chars, 14 bytes)
'truncated to 10': 'Zoë_Müller' (10 chars, 12 bytes)
fits in a 10-byte column? False

"Truncated to 10 characters" produced a string that's still 12 bytes — two bytes over the actual limit — purely because two of its characters (ë and ü) each cost 2 bytes in UTF-8 instead of 1.

Quick recap
  • Most data corruption bugs never raise an exception — the program runs, returns something plausible, and the only sign anything is wrong is that the value itself is incorrect.
  • Boundary math (off-by-one pagination, byte-vs-character limits, fixed-width overflow) is a recurring source of silent corruption precisely because each individual number involved looks reasonable in isolation.
  • Text encoding (issues #45, #48, #50) is a common, underestimated source of production bugs the moment real-world names, addresses, or free text enter a system that assumed plain ASCII.
In the Real World (Category 5)
  • Data corruption bugs are the ones that show up in "why don't these two reports match" tickets weeks or months after the code that caused them shipped — by the time anyone notices, the bad data may already be mixed into everything downstream.
  • Financial and e-commerce systems overwhelmingly use decimal.Decimal instead of float for money specifically because of issue #43 — floating-point rounding error is unacceptable when the numbers represent real currency.
  • "It passed code review" doesn't catch most of these bugs, because the code looks entirely reasonable line by line — catching pagination off-by-ones (#46) or mutable-default traps (#41) usually takes either a specific test case or having been burned by that exact bug before.
Category 6 of 8

6. Deployment & Environment

Every bug so far in this chapter can be reproduced by just running the code. #51

Using a language feature newer than production's Python version

Symptom

Code that works perfectly on a developer's machine fails to even start on the production server, with a SyntaxError or ModuleNotFoundError raised the instant the file is loaded — before any of the actual logic runs.

Root cause

Python's standard library and syntax both grow over time — structural pattern matching (match/case) arrived in 3.10; tomllib became part of the standard library only in 3.11. Code written and tested on a developer's up-to-date local Python has no way to know that a production server, provisioned from an older base image, is running a version from before that feature existed.

Fix

Pin the exact Python version in a .python-version file (or the container base image tag) and match it across every environment — local, CI, and production — so "works on my machine" and "works in production" are testing the same interpreter, not just the same source code.

#55

Symptom

A report scheduled to run "at 9am" fires at 9am on a developer's laptop during testing, but fires at a completely different real-world time once deployed to a production server.

Root cause

#61

SQL injection from string-formatted queries

Symptom

A user lookup that's supposed to return exactly one account instead returns every account in the entire table, for certain unusual-looking input.

Root cause

Chapter 21 covered running SQL from Python, but building the query with an f-string means the username value becomes literal SQL text, not data. An input like ' OR '1'='1 closes the string literal early and adds a condition that's always true — turning WHERE username = '' into WHERE username = '' OR '1'='1', which matches every row in the table. This is the single most well-known real-world security vulnerability class, and it comes directly from mixing code and data.

Fix

Always use parameterized queries — pass values as separate arguments (conn.execute(query, (username,))) rather than formatting them into the SQL string. The database driver handles escaping correctly, and a malicious value can never be interpreted as part of the SQL structure.

issue_61_sql_injection.py 
import sqlite3

conn = sqlite3.connect(":memory:")

conn.execute("INSERT INTO users VALUES (1, 'alice', 0)")
conn.execute("INSERT INTO users VALUES (2, 'bob', 1)")
conn.commit()

def find_user_vulnerable(username):
    # BUG: string formatting builds the SQL directly -- a malicious
    # username can inject its own SQL
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return conn.execute(query).fetchall()

# a normal lookup
print("normal lookup:", find_user_vulnerable("alice"))

# a malicious "username" that breaks out of the string literal

    # FIX: parameterized query -- the driver handles escaping, the value
    # is never concatenated into the SQL text itself

    return conn.execute(query, (username,)).fetchall()

#63

The N+1 query pattern

Symptom

A "list books with their authors" endpoint that's fast in testing with a handful of books becomes noticeably slow once the real database has more than a few hundred — and the slowdown is roughly proportional to the number of books, not the complexity of the query.

Root cause

This is Category 2's nested-loop pattern (issue #14), but expressed as database round-trips instead of in-memory list scans: one query fetches all the books, and then a separate query fetches each book's author, one at a time, inside a loop. For 150 books, that's 151 total queries — and every query is a full network round-trip to the database, which is far slower than any in-memory operation.

Fix

Use a JOIN to fetch books and their authors together in a single query — the database is specifically built to do this kind of combination efficiently, in one round-trip, rather than paying network latency once per row.

issue_63_n_plus_one.py 
import sqlite3

conn = sqlite3.connect(":memory:")
conn.execute("CREATE TABLE authors (id INTEGER, name TEXT)")
conn.execute("CREATE TABLE books (id INTEGER, title TEXT, author_id INTEGER)")
for i in range(1, 51):
    conn.execute("INSERT INTO authors VALUES (?, ?)", (i, f"Author {i}"))
    for j in range(3):
        conn.execute("INSERT INTO books VALUES (?, ?, ?)", (i * 10 + j, f"Book {i}-{j}", i))
conn.commit()

query_count = 0

def execute_counted(query, params=()):
    global query_count
    query_count += 1
    return conn.execute(query, params)

def list_books_with_authors_slow():
    # BUG: one query to get all books, then a SEPARATE query per book to
    # fetch its author -- the classic N+1 pattern
    books = execute_counted("SELECT id, title, author_id FROM books").fetchall()
    results = []
    for book_id, title, author_id in books:
        author = execute_counted("SELECT name FROM authors WHERE id = ?", (author_id,)).fetchone()
        results.append((title, author[0]))
    return results

results = list_books_with_authors_slow()
print(f"books returned: {len(results)}")
print(f"total queries executed: {query_count}  (1 + one-per-book)")

def list_books_with_authors_fast():
    # FIX: a single JOIN replaces the N+1 queries with exactly 1
    rows = execute_counted(
        "SELECT books.title, authors.name FROM books "
        "JOIN authors ON books.author_id = authors.id"
    ).fetchall()
    return rows

print(f"books returned (joined): {len(results_fast)}")

#64

Not handling a 429 rate-limit response

Symptom

A script that fetches several pages from a third-party API works fine for the first few pages, then crashes with a KeyError partway through — on a field that was present in every earlier response.

Root cause

Chapter 19 covered parsing a JSON response body, but assumed every response has the same shape. Once an API's rate limit is exceeded, it typically returns a 429 status with a completely different body (an error description, not the requested data) — code that reaches straight for response.json()["data"] without first checking the status code has no way to distinguish a successful response from a rate-limited one until the KeyError happens.

Fix

Check the status code before trusting the response body's shape, and implement a retry with exponential backoff specifically for 429s (many APIs also return a Retry-After header telling you exactly how long to wait) — rather than assuming every response that comes back is the one that was asked for.

#65

No timeout set on a network call, so it hangs forever

Symptom

A request to a downstream service occasionally never returns at all — not slow, not erroring, just permanently stuck — and every worker thread or process that gets stuck this way is effectively lost until the whole service is restarted.

Root cause

A network call with no timeout configured will wait indefinitely for a response, by design — that's the default behavior of raw sockets and most HTTP client libraries unless a timeout is explicitly requested. If the remote server is overloaded, or a firewall silently drops the packets rather than actively refusing the connection, there is nothing to make the call fail on its own; it just waits, forever, holding whatever resources (a thread, a connection) it was using.

Fix

Always set an explicit timeout on any network call — a deliberately failed request that can be retried or reported is always better than a silently hung one. Most production incidents involving a "stuck" service trace back to exactly this: one downstream call with no timeout, upstream of everything else.

#66

Retrying a non-idempotent request and double-charging a customer

Symptom

A payment retry mechanism that's supposed to make the system more reliable ends up charging a customer twice for a single purchase.

Root cause

Chapter 12 covered retrying operations after a failure — but retrying is only safe when the operation is idempotent, meaning doing it twice has the same effect as doing it once. A payment charge is not naturally idempotent: if the server-side charge succeeded but the response was lost on the way back (a timeout, a dropped connection), the client has no way to know that, and blindly retrying the same request charges the customer a second time for what the client believes is still a single failed attempt.

Fix

Attach a unique idempotency key to the request (generated once per logical operation, reused on every retry) so the server can recognize "I've already processed this exact request" and return the original result instead of charging again — most real payment APIs (Stripe, for example) support this pattern natively.

#70

Treating "the API call failed" as "the resource doesn't exist"

Symptom

Every product shows as "out of stock" for several minutes, across the entire catalog, even though inventory levels haven't actually changed — then it resolves on its own with no code deployed.

Root cause

This is issue #6's broad-except pattern, applied to a dependency instead of a business-logic exception.   • SQL injection (issue #61) and unhandled network hangs (issue #65) are two of the most consequential bugs in this entire chapter — one is a direct security vulnerability, the other can take down an entire service by exhausting its available workers.
  • A resource acquired from a pool (a connection) needs an equally explicit release, matching Category 3's file-handle lesson — forgetting either one eventually exhausts a fixed-size resource.

In the Real World (Category 7)
  • SQL injection remains, decades after it was first identified, one of the most common vulnerabilities found in real security audits — not because the fix is unknown, but because one unparameterized query anywhere in a large codebase is enough.

#71

print() statements instead of structured logging

Symptom

A developer insists a piece of code definitely ran and logged useful diagnostic info — but nothing matching it can be found anywhere in the production log aggregator.

Root cause

print() writes to standard output; the logging module writes through a configurable pipeline of handlers, levels, and formatters. Fix

Use the logging module everywhere in application code, never print(), so every message flows through one consistent, configurable pipeline — and explicitly configure the logging level appropriate for each environment (more verbose in development, less in production) rather than relying on the library's default.

#76

Symptom

A log line correctly reports that a row failed to import, and even shows the error message — but nobody can tell which line of code actually raised it, because a nearly identical ValueError message could come from three different places in the codebase.

Root cause

str(e) converts an exception to just its message text — everything else about the exception (which file, which line, the full call stack that led there) is discarded. logger.error(str(e)) logs only that bare message, which is often not unique or specific enough to pinpoint the actual failure once there's more than one place in the codebase that could raise a similarly worded error.

Fix

Use logger.exception(message) instead, called from inside an except block — it automatically attaches the full traceback to the log record, giving anyone reading the log the exact same information they'd have seen if the exception had crashed the program uncaught.

issue_76_error_str_not_exception.py 
import logging
import io

log_stream = io.StringIO()
handler = logging.StreamHandler(log_stream)
handler.setFormatter(logging.Formatter("%(levelname)s: %(message)s"))
logger = logging.getLogger("importer")
logger.addHandler(handler)
logger.setLevel(logging.INFO)

def parse_row(row):
    return int(row["amount"])  # raises if "amount" is missing or non-numeric

def import_rows(rows):
    for row in rows:
        try:
            parse_row(row)
        except Exception as e:
            # BUG: log.error(str(e)) discards the traceback entirely --
            # you get the error MESSAGE but no idea which line/function raised it
            logger.error(str(e))

import_rows([{"amount": "not-a-number"}])

print("--- with log.error(str(e)) ---")
print(log_stream.getvalue())
actual output (Python 3.14.4) 
--- with log.error(str(e)) ---
ERROR: invalid literal for int() with base 10: 'not-a-number'

Switching to logger.exception("failed to parse row") instead produces dramatically more useful output from the exact same failure:

actual output with log.exception() (Python 3.14.4) 
ERROR: failed to parse row
Traceback (most recent call last):
  File "issue_76_error_str_not_exception.py", line 38, in import_rows_fixed
    parse_row(row)
    ~~~~~~~~~^^^^^
  File "issue_76_error_str_not_exception.py", line 12, in parse_row
    return int(row["amount"])  # raises if "amount" is missing or non-numeric
ValueError: invalid literal for int() with base 10: 'not-a-number'

Same underlying failure, dramatically more information — the fixed version names the exact function and line (parse_row, then the specific int(...) call) that raised the error, not just what it said.

#77

Eagerly building an expensive log message that's never used

Symptom

A service is measurably slower than expected, even in production where debug logging is fully disabled — profiling points squarely at a function that's supposedly only used for debug output.

Root cause

An f-string is evaluated the instant Python reaches that line — logger.debug(f"...{expensive_summary(data)}") calls expensive_summary(data) immediately, building the entire formatted string, before logger.debug() even gets a chance to decide whether the message will actually be used. If the logger's level is above DEBUG (the common case in production), all of that formatting work happens for absolutely nothing — the result is computed and then immediately discarded.

Fix

For genuinely expensive debug-only computations, guard them with if logger.isEnabledFor(logging.DEBUG): so the expensive work only happens when the message will actually be used. (For ordinary log arguments, logging's built-in %-style lazy formatting — logger.debug("value: %s", x) — already defers formatting; it's specifically an expensive function call embedded in an eager f-string that needs this extra guard.)

issue_77_fstring_eager_eval.py 
import logging
import io
import time

log_stream = io.StringIO()
handler = logging.StreamHandler(log_stream)
logger = logging.getLogger("perf")
logger.addHandler(handler)
logger.setLevel(logging.WARNING)  # DEBUG-level messages are disabled

call_count = 0

def expensive_summary(data):
    global call_count
    call_count += 1  # stands in for genuinely expensive formatting work
    return f"summary of {len(data)} records: {sum(data)}"

data = list(range(1000))

# BUG: an f-string is built EAGERLY, every time this line runs, even
# though logger.debug() will immediately throw the result away because
# the logger is set to WARNING
logger.debug(f"debug info: {expensive_summary(data)}")
print(f"expensive_summary() was called even though nothing was logged: {call_count} time(s)")

call_count = 0
# FIX: pass the expensive part as a lazy argument -- logging only calls
# str() on it (which triggers the % formatting) if the message will
# actually be emitted
logger.debug("debug info: %s", expensive_summary(data) if logger.isEnabledFor(logging.DEBUG) else None)
print(f"with an isEnabledFor() guard, expensive_summary() was called: {call_count} time(s)")
actual output (Python 3.14.4) 
expensive_summary() was called even though nothing was logged: 1 time(s)
with an isEnabledFor() guard, expensive_summary() was called: 0 time(s)
#78

A log file that grows forever and eventually fills the disk

Symptom

A service that's been running for weeks suddenly starts throwing disk-related errors on completely unrelated operations — writing a temp file, saving an upload — all because the disk itself is full.

Root cause

A plain logging.FileHandler opens one file and keeps appending to it, with no concept of a size limit — there's nothing built into it that ever rotates, truncates, or deletes old content. For a service that logs continuously over weeks or months of uptime, that one file grows without bound until it eventually consumes all available disk space, at which point every disk-writing operation on the whole machine starts failing, not just logging.

Fix

Use logging.handlers.RotatingFileHandler (or TimedRotatingFileHandler) instead of a plain FileHandler — it automatically rotates to a new file once the current one hits a size limit, and deletes the oldest backups once a configured number of backups is exceeded, capping total disk usage.

issue_78_unbounded_log_growth.py 
import logging
import os

log_path = "/tmp/ch26_app.log"
if os.path.exists(log_path):
    os.remove(log_path)

logger = logging.getLogger("noisy")
logger.setLevel(logging.INFO)
# BUG: a plain FileHandler never rotates or caps the file -- it just
# keeps appending forever, until disk fills up
handler = logging.FileHandler(log_path)
logger.addHandler(handler)

for i in range(5000):
    logger.info(f"processed item {i}" * 3)  # simulate a somewhat verbose line

handler.flush()
size_bytes = os.path.getsize(log_path)
print(f"log file size after 5,000 lines: {size_bytes:,} bytes")
print("a plain FileHandler has no size cap -- this keeps growing indefinitely,")
print("which is exactly how a busy service fills its disk over weeks of uptime")

os.remove(log_path)
actual output (Python 3.14.4) 
log file size after 5,000 lines: 286,670 bytes
a plain FileHandler has no size cap -- this keeps growing indefinitely,
which is exactly how a busy service fills its disk over weeks of uptime

Swapping in RotatingFileHandler(log_path, maxBytes=50_000, backupCount=3) for the exact same 5,000 log calls caps total disk usage at roughly 4 files of 50KB each, automatically discarding anything older than the 3 most recent backups — instead of one file that would otherwise keep growing without limit.

#79

A silently swallowed exception with no log line at all

Symptom

A customer calls support to say they never received an order confirmation email. There is absolutely nothing in any log, anywhere, that mentions that customer, that order, or any failure at all.

Root cause

This is issue #3's bare except: pattern taken to its most damaging extreme: except Exception: pass doesn't just catch too broadly — it produces zero record of the failure ever happening. Even a badly written log line beats no log line at all; with nothing logged, there is no starting point from which to even begin investigating, because there's no evidence the failure occurred in the first place.

Fix

Never write a bare except: pass (or except Exception: pass) in production code — at an absolute minimum, log the exception (with logger.exception(), per issue #76) even in a code path that's deliberately designed to continue past a failure, so there's a permanent record that something went wrong, even if the immediate decision is not to stop and surface it to the caller.

issue_79_exception_swallowed_no_log.py 
def send_notification(user_id, message):
    if user_id == "user-999":
        raise ConnectionError("notification service unreachable")
    return "sent"

def notify_all(user_ids, message):
    sent = 0
    for uid in user_ids:
        try:
            send_notification(uid, message)
            sent += 1
        except Exception:
            pass  # BUG: completely silent -- not even a log line, nothing
    return sent

result = notify_all(["user-1", "user-999", "user-3"], "Your order shipped!")
print(f"notify_all() reports {result} of 3 notifications sent")
print("one silently failed -- there is NO record anywhere that it happened,")
print("no log line, no metric, nothing an on-call engineer could ever find")
actual output (Python 3.14.4) 
notify_all() reports 2 of 3 notifications sent
one silently failed -- there is NO record anywhere that it happened,
no log line, no metric, nothing an on-call engineer could ever find

The function's own return value (2 out of 3) is the only hint anything went wrong — and only if the caller happens to check it and knows to compare it against the number of users passed in. Nothing forces that comparison to happen.

#80

Unstructured log messages that can't be queried at scale

Symptom

Answering "how many orders over $500 failed last week" requires someone to write a fragile regular expression against raw log text, rather than a simple, direct query.

Root cause

A free-form log message like f"Order {order_id} for customer {customer_id}: ${amount} - {status}" is easy for a human to read in isolation, but a log aggregator has no idea that the second number is a dollar amount or that the last word is a status — to it, it's just an unstructured string. Extracting any specific field back out means writing a regex that has to be updated every time the message's wording changes even slightly.

Fix

Log structured data — typically JSON — so each field is explicitly named and typed: {"order_id": ..., "amount": ..., "status": ...}. Nearly every modern log aggregator can index structured fields directly, turning "how many failed orders over $500" from a regex-writing exercise into a straightforward filter.

issue_80_unstructured_vs_structured_log.py 
import logging
import io
import json

log_stream = io.StringIO()
handler = logging.StreamHandler(log_stream)
handler.setFormatter(logging.Formatter("%(message)s"))
logger = logging.getLogger("orders_unstructured")
logger.addHandler(handler)
logger.setLevel(logging.INFO)

def log_order_unstructured(order_id, customer_id, amount, status):
    # BUG: a free-form string -- easy to write, hard to query at scale.
    # Finding "every failed order over $500" means regex-parsing this text.
    logger.info(f"Order {order_id} for customer {customer_id}: ${amount} - {status}")

log_order_unstructured("ord-1", "cust-9", 750, "failed")

print("--- unstructured ---")
print(log_stream.getvalue())

log_stream2 = io.StringIO()
handler2 = logging.StreamHandler(log_stream2)
handler2.setFormatter(logging.Formatter("%(message)s"))
logger2 = logging.getLogger("orders_structured")
logger2.addHandler(handler2)
logger2.setLevel(logging.INFO)

def log_order_structured(order_id, customer_id, amount, status):
    # FIX: a structured (JSON) log line -- trivially queryable by any log
    # aggregator: "amount > 500 AND status = failed", no regex required
    logger2.info(json.dumps({
        "order_id": order_id,
        "customer_id": customer_id,
        "amount": amount,
        "status": status,
    }))

log_order_structured("ord-1", "cust-9", 750, "failed")

print("--- structured (JSON) ---")
print(log_stream2.getvalue())
actual output (Python 3.14.4) 
--- unstructured ---
Order ord-1 for customer cust-9: $750 - failed

--- structured (JSON) ---
{"order_id": "ord-1", "customer_id": "cust-9", "amount": 750, "status": "failed"}

Thought process: structured logging is a decision that pays off precisely when it matters most — during an incident, under time pressure, when someone needs a specific answer from thousands of log lines fast. Its cost (writing json.dumps({...}) instead of an f-string) is trivial and paid once; its benefit compounds every single time anyone needs to query the logs afterward.

Quick recap
  • Nearly every bug in this category shares one theme: the evidence that would have explained an incident was never captured, either because it was never logged at all (issues #71, #79) or because it was logged in a form that turned out to be useless later (issues #72, #76, #80).
  • Logging a caught exception is not the same as handling it (issue #75) — a caller or downstream system still needs to know the operation failed, through a return value or a re-raised exception, not just through a log line a human might eventually read.
  • Good observability is mostly about anticipating the question someone will ask during an incident (which order, which customer, did this even run) and making sure the logs can already answer it, rather than trying to add the right log line after the incident has already happened.
In the Real World (Category 8)
  • "Mean time to resolution" during a real incident is overwhelmingly determined by how good the logs are, not by how skilled the engineer on call is — a well-instrumented system can be debugged by almost anyone; a poorly logged one can stump an expert for hours.
  • Structured logging (issue #80) combined with correlation IDs (issue #73) is the foundation that modern distributed tracing tools (like OpenTelemetry) build on top of — getting these two habits right early makes adopting more sophisticated observability tooling later far easier.
  • Logging secrets (issue #72) is one of the most common findings in real security audits, and one of the most embarrassing, precisely because the fix (don't log sensitive fields) is simple in hindsight but easy to miss in the moment while debugging something else entirely.

Points to Remember

  • KeyError, IndexError, TypeError, AttributeError, ZeroDivisionError, RecursionError, and JSONDecodeError each have a specific, predictable trigger — recognizing the pattern is faster than reading the whole traceback from scratch every time.
  • Bare except: and overly broad except Exception: are the two patterns most likely to turn a real bug into a silent, wrong-but-plausible result — across crashes (#3, #6), memory leaks left uncapped, and dependencies failing silently (#70), this single habit shows up again and again as the actual root cause.
  • The most dangerous production bugs are rarely the loud ones — a crash is at least visible. Silent wrong answers (mutable default arguments, shallow copies, off-by-one pagination, float equality, an absorbed exception) can sit undetected in a system for months.
  • Unbounded growth is a recurring shape across totally different bugs: an ever-growing cache or list (Category 3), a log file with no rotation (#78), a connection pool nobody returns to (#62) — the fix is almost always the same idea, a deliberate limit, applied in a different place.
  • Concurrency bugs (races, deadlocks) and environment bugs (version mismatches, missing config) share a frustrating trait: they can pass every test run and still be genuinely broken, because the bug depends on timing or on a difference between machines that testing didn't happen to exercise.
  • Good observability means anticipating, before an incident happens, the specific question someone will ask while debugging it (which order, which customer, did this even run) — structured logs, correlation IDs, and full tracebacks (logger.exception(), not str(e)) are what make that question answerable later.
