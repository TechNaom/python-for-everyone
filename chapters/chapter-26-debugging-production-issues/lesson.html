#1

KeyError: 'DATABASE_URL'

Symptom

The service crashes on startup, before it ever handles a single request, with a KeyError pointing at a config key.

Root cause

Chapter 8 covered that some_dict[key] raises KeyError the instant the key isn't present — there's no "closest match" or silent fallback. A hardcoded config dict is missing a key that a deployment step (an environment variable, a secrets file) was supposed to provide.

Fix

Use config.get("DATABASE_URL") with an explicit fallback or a clear startup-time error message, instead of letting a raw KeyError (which doesn't say why the key is missing) surface as the crash.

issue_01_keyerror.py 
config = {"HOST": "localhost", "PORT": 5432}

def connect():
    db_url = config["DATABASE_URL"]
    return f"Connecting to {db_url}"

print(connect())

#2

IndexError: list index out of range

Symptom

A report-generation function crashes intermittently — it works for most inputs, then suddenly fails with IndexError once a list is shorter than expected.

Root cause

Chapter 5 covered that range(n) doesn't know or care how long any particular list is — this function assumes there are always at least n items to loop over, and that assumption silently holds until the day the list is shorter.

Fix

Bound the loop by the list's own length: range(min(n, len(scores))), or loop over scores[:n] directly so the loop can never ask for an index that isn't there.

issue_02_indexerror.py 
scores = [88, 92, 79, 95]

def bonus_for_top_n(scores, n):
    total = 0
    for i in range(n):
        total += scores[i]
    return total

print(bonus_for_top_n(scores, 5))

#3

A bare except: silently hides a typo, forever

Symptom

A settings loader "works" — it never crashes, and it always returns a default timeout of 30. Except it's always returning the default, even for configs where a real value was actually provided, and nobody notices for months.

Root cause

Chapter 12 covered that a bare except: catches every exception, not just the one you had in mind. Here the config key is typo'd ("timeoout"), which raises a KeyError — and the bare except: quietly swallows it and returns the fallback, exactly as if the key had simply been left out on purpose. The typo becomes invisible.

Fix

Catch the specific exception you actually expect (except KeyError:), never a bare except: — that way a genuine typo or unrelated bug still surfaces as its own distinct crash instead of being silently absorbed into "using the default."

issue_03_bare_except.py 
def load_settings(raw):
    try:
        return int(raw["timeout"])
    except:
        return 30

# a typo in the config key -- "timeoout", not "timeout"
print(load_settings({"timeoout": "45"}))
actual output (Python 3.14.4) 
30

Nothing here looks broken — the function runs, returns a plausible number, and never raises. That's exactly the danger: change except: to except KeyError: and run it again, and the real bug (a typo'd key that means the configured timeout of 45 is never actually used) finally surfaces as a visible KeyError instead of hiding behind a fallback value forever.

#4

TypeError: unsupported operand type(s) for /: 'str' and 'int'

Symptom

A checkout function crashes with a TypeError several function calls deep — the traceback points at a line of ordinary-looking arithmetic.

Root cause

Chapter 2 covered that Python is dynamically typed — nothing stops a caller from passing a string where a number was expected. Here percent_off arrives as the string "10" (likely straight from form input or a query parameter) and gets passed through two function calls before the arithmetic finally fails, far from where the bad value actually originated.

Fix

Convert and validate external input at the boundary where it enters the system (right after reading it from a form/request), not deep inside business logic — by the time apply_discount() runs, percent_off should already be guaranteed numeric.

issue_04_typeerror.py 
def apply_discount(price, percent_off):
    return price - (price * percent_off / 100)

def checkout_total(price, percent_off):
    return apply_discount(price, percent_off)

# percent_off arrived as a string, e.g. straight from request.form
print(checkout_total(100, "10"))

#5

AttributeError: 'NoneType' object has no attribute 'get'

Symptom

Sending a welcome email crashes with an AttributeError mentioning NoneType — a type that was never mentioned anywhere in the code that crashed.

Root cause

Chapter 10 covered that a function with no explicit return on some code path implicitly returns None on that path. find_user() only returns inside the loop when a match is found — if the loop finishes without matching, the function falls through and returns None, with nothing announcing that it happened.

Fix

Check for None right where the value comes back (if user is None: ...) before calling any method on it — or raise a clear, named exception from find_user() itself when no match exists, instead of returning None silently.

issue_05_attributeerror.py 
users = [
    {"name": "Ana", "email": "ana@example.com"},
    {"name": "Beto"},
]

def find_user(name):
    for u in users:
        if u["name"] == name:
            return u
    # falls through here if nothing matched -- implicitly returns None

def send_welcome_email(name):
    user = find_user(name)
    return user.get("email").upper()

print(send_welcome_email("Carla"))

#6

A custom exception gets raised correctly -- and swallowed anyway

Symptom

A withdrawal of $200 from a $50 balance is reported as successful, with the balance left unchanged at $50 — no error, no warning, and no money actually moved (which is the only reason this bug is survivable at all).

Root cause

Chapter 12 covered writing custom exception classes specifically so callers can react to meaningful, named failures. InsufficientFundsError is raised correctly here — but the caller's except Exception: is broad enough to catch it too, right alongside genuine bugs, and silently falls back to "return the balance unchanged" as if nothing happened.

Fix

Catch the specific custom exception (except InsufficientFundsError:) and handle it deliberately — show the user a real "insufficient funds" message — instead of catching Exception broadly and treating every possible failure the same way.

issue_06_swallowed_custom_exception.py 
class InsufficientFundsError(Exception):
    pass

def withdraw(balance, amount):
    if amount > balance:
        raise InsufficientFundsError(f"Cannot withdraw {amount}, balance is {balance}")
    return balance - amount

def process_withdrawal(balance, amount):
    try:
        return withdraw(balance, amount)
    except Exception:
        return balance

result = process_withdrawal(50, 200)
print(f"Balance after withdrawal attempt: {result}")
actual output (Python 3.14.4) 
Balance after withdrawal attempt: 50

The output looks completely reasonable in isolation — it's only wrong once you know a $200 withdrawal was silently rejected instead of clearly refused. This is the same shape of bug as #3: a too-broad except turning a real, specific failure into indistinguishable "normal" behavior.

#7

RecursionError: maximum recursion depth exceeded

Symptom

A recursive countdown function crashes with a huge traceback ending in RecursionError, instead of just counting down and stopping.

Root cause

Chapter 10 covered that every recursive function needs a base case that's actually reachable. countdown()'s base case checks n != 0 — but called with a negative starting number, n counts -5, -6, -7, ... and steps straight past 0 without ever landing exactly on it, so the base case never triggers.

Fix

Use a base case that can't be skipped over, such as n <= 0 instead of n != 0 — or validate the input and reject a negative starting value before recursion begins at all.

issue_07_recursionerror.py 
def countdown(n):
    print(n)
    if n != 0:
        countdown(n - 1)

countdown(-5)

#8

ZeroDivisionError: division by zero

Symptom

Computing a product's average rating crashes with ZeroDivisionError — but only for products that don't have any reviews yet.

Root cause

Chapter 3 covered that / raises ZeroDivisionError whenever the denominator is exactly 0 — no rounding, no special case. num_reviews here comes from a data source (this stands in for a real API/database result, tying back to Chapter 19) that's a perfectly valid 0 for a brand-new product, but the function never checks for it.

Fix

Check the denominator before dividing and return a sensible value for the zero case — 0, None, or "No reviews yet", whichever fits how the result gets displayed — instead of letting the division itself decide by crashing.

def average_rating(total_score, num_reviews):

# a brand-new product with no reviews yet
review_data = {"total_score": 47, "num_reviews": 0}

#9

FileNotFoundError and PermissionError look similar -- and aren't

Symptom

A log reader was written to handle "the file doesn't exist yet" gracefully. In production it crashes anyway — not because the file is missing, but because the process doesn't have permission to read it.

Root cause

Chapter 13 covered handling FileNotFoundError around open(). That's a different, more specific exception than PermissionError (raised when the file exists but the OS denies access — a locked-down production deploy directory, for example). Code that only catches FileNotFoundError lets a real PermissionError propagate as an unhandled crash, and the two are easy to mentally lump together as "file problems" even though the fix for each is completely different.

Fix

Catch both explicitly, and handle them differently: FileNotFoundError usually means "treat this as empty/first run"; PermissionError usually means the deployment itself is misconfigured and should fail loudly with a message an operator can act on, not be silently absorbed.

issue_09_file_errors.py 
def read_log(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        return "no log file yet"

print(read_log("/var/log/myapp/nonexistent-2026-07-09.log"))
actual output (Python 3.14.4) 
no log file yet

This particular run succeeds because the path genuinely doesn't exist. The bug only shows up in an environment where the path exists but is locked down — at that point, open() raises PermissionError instead, which this except FileNotFoundError: does not catch, and the crash propagates uncaught.

#10

json.decoder.JSONDecodeError: Expecting value

Symptom

Parsing an API response crashes with JSONDecodeError — not because the API sent malformed JSON, but because it didn't send JSON at all.

Root cause

Chapter 19 covered json.loads() and checking a response's status code before trusting its body. This code skips that check and assumes every response body is valid JSON — but a real outage, rate limit, or misconfigured URL often returns an HTML error page (a login redirect, a "503 Service Unavailable" page) instead, and json.loads() has no way to parse that as JSON.

Fix

Check response.status_code (or equivalent) before parsing, and wrap json.loads() in its own try/except json.JSONDecodeError so a non-JSON response produces a clear "the API returned something unexpected" error instead of a raw parser traceback.

issue_10_jsondecodeerror.py 
import json

def parse_api_response(raw_text):
    data = json.loads(raw_text)
    return data["results"]

# what a downed API actually returned -- an HTML error page, not JSON
fake_response_body = "<html><body>Service Unavailable</body></html>"
print(parse_api_response(fake_response_body))

Quick recap
  • Most production crashes trace back to an assumption that was true in testing and false in the real world: a key that's always present, a list that's always long enough, input that's always the right type, a denominator that's never zero.
  • A bare except: or an overly broad except Exception: doesn't prevent bugs — it hides them, often turning a loud, easy-to-diagnose crash into a silent wrong answer (issues #3 and #6).
  • Similarly-named exceptions (FileNotFoundError vs. PermissionError) can require genuinely different handling — catching one doesn't mean the other is covered.
In the Real World (Category 1)
  • Production incident write-ups almost always follow this exact shape — symptom, root cause, fix — because that's the order information actually becomes available: you see the crash first, then dig into why, then decide what changes.
  • The most dangerous bugs in real systems are rarely the ones that crash loudly (issues #1, #2, #4, #5, #7, #8, #10 here) — they're the ones that silently return a plausible-but-wrong answer (issues #3 and #6), because nothing alerts anyone that something went wrong.
  • Being able to predict which exception a piece of code is about to throw — before running it — is a skill senior engineers use constantly during code review, and it's built entirely from knowing these patterns cold.

#11

Checking membership against a growing list, over and over

Symptom

A "find duplicates" function that's fast on a small test list becomes noticeably slow once the real dataset has a few thousand items — and gets dramatically worse as the dataset grows, not just proportionally worse.

Root cause

Chapter 8 covered that checking item in some_list has to scan the list from the start until it finds a match (or reaches the end) — an O(n) operation. Doing that check inside a loop that runs n times makes the whole function O(n²): every new item re-scans an ever-growing list.

Fix

Use a set instead of a list for the membership check — Chapter 8 also covered that item in some_set is O(1) on average, regardless of how large the set gets.

issue_11_quadratic_membership.py 
import time

def find_duplicates_slow(items):
    seen = []
    duplicates = []
    for item in items:
        if item in seen:  # O(n) list scan, called n times -> O(n^2)
            duplicates.append(item)
        else:
            seen.append(item)
    return duplicates

def find_duplicates_fast(items):
    seen = set()
    duplicates = []
    for item in items:
        if item in seen:  # O(1) average-case set membership
            duplicates.append(item)
        else:
            seen.add(item)
    return duplicates

data = list(range(5000)) + list(range(2000))

start = time.perf_counter()
find_duplicates_slow(data)
slow_time = time.perf_counter() - start

start = time.perf_counter()
find_duplicates_fast(data)
fast_time = time.perf_counter() - start

print(f"list-based (O(n^2)): {slow_time:.4f}s")
print(f"set-based (O(n)):    {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
list-based (O(n^2)): 0.0972s
set-based (O(n)):    0.0008s

Thought process: the actual fix here isn't "sets are faster than lists" as a blanket rule — it's recognizing the specific access pattern (repeated membership checks against a growing collection) where the difference between O(1) and O(n) per check compounds into O(n) vs. O(n²) overall. An experienced dev looks for this exact shape: any in check inside a loop, against a collection that's also being built up inside that same loop.

#12

Inserting at the front of a list in a loop

Symptom

A function that reverses the order of a list by building it up one item at a time gets dramatically slower as the input grows, even though it's doing the same amount of "work" per item.

Root cause

Chapter 7 covered that list.insert(0, x) has to shift every existing element over by one to make room at the front — an O(n) operation. Calling it n times to build an n-item list makes the whole function O(n²), the same shape of problem as issue #11 but from a different list operation.

Fix

list.append() adds to the end in O(1) amortized time. Append everything in order, then call .reverse() once at the very end — one O(n) pass total, instead of n separate O(n) shifts.

issue_12_insert_front.py 
import time

def reverse_order_slow(items):
    result = []
    for item in items:
        result.insert(0, item)  # O(n) shift, every single call
    return result

def reverse_order_fast(items):
    result = []
    for item in items:
        result.append(item)  # O(1) amortized
    result.reverse()  # one O(n) pass at the end
    return result

items = list(range(20000))

start = time.perf_counter()
a = reverse_order_slow(items)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = reverse_order_fast(items)
fast_time = time.perf_counter() - start

print(a == b)
print(f"insert(0, x) in a loop: {slow_time:.4f}s")
print(f"append then reverse():  {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine

append then reverse():  0.0005s
#13

Recomputing the same expensive result over and over

Symptom

A checkout function that calculates tax for a shopping cart is slow specifically when the cart has many items from the same handful of regions — even though there's only a few distinct regions actually involved.

Root cause

Chapter 10 covered function calls as the unit of reusable work — but nothing about a function call automatically remembers a previous result. A tax-rate lookup was called once per cart item, even though the same region's rate was computed identically, from scratch, every single time it recurred.

Fix

Cache results in a dict (Chapter 8) keyed by the input that determines the result — here, the region. Look up the cache before doing the expensive work, and only do the work (and store it) the first time a given region is seen.

issue_13_repeated_recompute.py 
import time

def slow_tax_lookup(region):
    # stands in for a genuinely expensive computation
    total = 0
    for _ in range(20000):
        total += 1
    return 0.08

def total_price_slow(cart, get_tax_rate):
    total = 0
    for item in cart:
        rate = get_tax_rate(item["region"])  # recomputed for every item
        total += item["price"] * (1 + rate)
    return total

def total_price_fast(cart, get_tax_rate):
    total = 0
    cache = {}
    for item in cart:
        region = item["region"]
        if region not in cache:
            cache[region] = get_tax_rate(region)
        total += item["price"] * (1 + cache[region])
    return total

cart = [{"price": 10, "region": "CA"} for _ in range(500)]

start = time.perf_counter()
total_price_slow(cart, slow_tax_lookup)
slow_time = time.perf_counter() - start

start = time.perf_counter()
total_price_fast(cart, slow_tax_lookup)
fast_time = time.perf_counter() - start

print(f"recompute every item: {slow_time:.4f}s")
print(f"cache by region:      {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
recompute every item: 0.3718s
cache by region:      0.0007s

Thought process: caching is only a correct fix when the same input reliably produces the same output — the region's tax rate doesn't change mid-checkout, so caching by region is safe here. An experienced dev checks that assumption explicitly before reaching for a cache, since caching a result that legitimately changes between calls (a live stock price, a random value) would introduce a correctness bug, not just a performance win.

#14

A nested loop to join two lists of records

Symptom

A function that attaches customer names to a list of orders works fine in testing with a handful of records, then times out once it's run against the real orders and customers tables.

Root cause

For every order, the code scans the entire customers list looking for a matching ID — an O(m) scan repeated n times, for O(n·m) overall. With a few thousand of each, that's tens of millions of comparisons for what should be a simple lookup.

Fix

Build a dict mapping customer ID to customer record once, up front (Chapter 8's dict/set-comprehension-friendly pattern), then look up each order's customer in O(1) instead of scanning — O(n + m) overall instead of O(n·m).

issue_14_nested_loop_lookup.py 
import time

def enrich_orders_slow(orders, customers):
    enriched = []
    for order in orders:
        for customer in customers:  # full scan, for EVERY order
            if customer["id"] == order["customer_id"]:
                enriched.append({**order, "customer_name": customer["name"]})
                break
    return enriched

def enrich_orders_fast(orders, customers):
    customer_by_id = {c["id"]: c for c in customers}  # built once
    enriched = []
    for order in orders:
        customer = customer_by_id.get(order["customer_id"])
        enriched.append({**order, "customer_name": customer["name"] if customer else None})
    return enriched

customers = [{"id": i, "name": f"Customer {i}"} for i in range(2000)]
orders = [{"customer_id": i % 2000, "amount": 10} for i in range(2000)]

start = time.perf_counter()
a = enrich_orders_slow(orders, customers)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = enrich_orders_fast(orders, customers)
fast_time = time.perf_counter() - start

print(a == b)
print(f"nested loop (O(n*m)): {slow_time:.4f}s")

#15

Recompiling the same regular expression on every call

Symptom

A log-scanning function using re.search() is a bit slower than expected on a large batch of lines — a real but modest overhead, not a dramatic one.

Root cause

Chapter 18 covered re.search(pattern, text) as a convenience function — internally, it still has to turn the pattern string into a compiled matcher before it can search. Called repeatedly with the same pattern string, that parsing step repeats every time (CPython does cache a small number of recently-used patterns internally, which is why this speedup is real but modest rather than dramatic).

Fix

Call re.compile(pattern) once, outside the loop, and call .search()/.match() on the resulting compiled pattern object instead — the parsing step happens exactly once no matter how many times it's used afterward.

issue_15_regex_recompile.py 
import re
import time

PATTERN_STR = r"\d{3}-\d{3}-\d{4}"

def find_phone_numbers_slow(lines):
    found = []
    for line in lines:
        match = re.search(PATTERN_STR, line)  # parses PATTERN_STR every call
        if match:
            found.append(match.group())
    return found

def find_phone_numbers_fast(lines):
    compiled = re.compile(PATTERN_STR)  # parsed once
    found = []
    for line in lines:
        match = compiled.search(line)
        if match:
            found.append(match.group())
    return found

lines = [f"Contact: 555-{i:03d}-{i:04d} for details" for i in range(30000)]

# take the BEST of 5 runs for each version -- a single run's timing can be
# noisy (one-off system interference); best-of-several is more stable
def timed(fn, arg):
    start = time.perf_counter()
    result = fn(arg)
    return time.perf_counter() - start, result

slow_time, a = min((timed(find_phone_numbers_slow, lines) for _ in range(5)), key=lambda t: t[0])
fast_time, b = min((timed(find_phone_numbers_fast, lines) for _ in range(5)), key=lambda t: t[0])

print(a == b)
print(f"re.search() each time (best of 5): {slow_time:.4f}s")
print(f"re.compile() once (best of 5):     {fast_time:.4f}s")
actual output (Python 3.14.4, best of 5 runs each) -- your own numbers will vary by machine 
True
re.search() each time (best of 5): 0.0267s
re.compile() once (best of 5):     0.0205s

A real speedup, but a modest one — not every fix in this category is a dramatic win, and it's worth being honest about which is which. re.compile() is still worth doing as a habit: it costs nothing, and the margin grows if the pattern is more complex than this one.

#16

Sorting an entire list just to read the largest value

Symptom

A "top score" function is noticeably slower than it needs to be on a large dataset, despite only ever using one value out of the whole result.

Root cause

sorted() is O(n log n) — it does the work of putting every element in order, even when the only thing the caller actually wants is the single largest one.

Fix

max() finds the largest value in a single O(n) pass, without sorting anything. Use the built-in that matches what's actually needed — max()/min() for one extreme value, sorted() only when the full order genuinely matters.

issue_16_sort_for_max.py 
import time
import random

def top_score_slow(scores):
    ordered = sorted(scores)  # O(n log n) to sort everything...
    return ordered[-1]         # ...just to read the last element

def top_score_fast(scores):
    return max(scores)  # O(n)

scores = list(range(200000))
random.shuffle(scores)

start = time.perf_counter()
a = top_score_slow(scores)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = top_score_fast(scores)
fast_time = time.perf_counter() - start

print(a == b)
print(f"sorted()[-1]: {slow_time:.4f}s")
print(f"max():        {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
True
sorted()[-1]: 0.0704s
max():        0.0092s
#17

Re-evaluating len() on every loop iteration

Symptom

A hand-written while loop summing a very large list is measurably slower than the equivalent for loop, even though both do the same O(n) amount of "real" work.

Root cause

This one isn't about Big-O — len() on a list is already O(1). It's about repeated small overhead: while i < len(items): calls len() fresh on every single iteration, and that function-call overhead adds up across millions of iterations even though each individual call is cheap.

Fix

Compute len(items) once, store it in a variable before the loop starts, and check against that variable instead — the list's length doesn't change during the loop here, so there's nothing to recompute.

issue_17_len_in_condition.py 
import time

def sum_while_slow(items):
    total = 0
    i = 0
    while i < len(items):  # len() called fresh every iteration
        total += items[i]
        i += 1
    return total

def sum_while_fast(items):
    total = 0
    i = 0
    n = len(items)  # computed once
    while i < n:
        total += items[i]
        i += 1
    return total

items = list(range(2000000))

start = time.perf_counter()
a = sum_while_slow(items)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = sum_while_fast(items)
fast_time = time.perf_counter() - start

print(a == b)
print(f"len(items) every iteration: {slow_time:.4f}s")
print(f"len(items) computed once:   {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
True
len(items) every iteration: 0.2065s
len(items) computed once:   0.1292s

Another honest, modest speedup — and a preview of why Chapter 5's plain for item in items: is usually preferable to a hand-rolled while loop with a manual index in the first place: it sidesteps this whole class of micro-overhead automatically.

#18

Re-reading the same file from disk, once per search term

Symptom

A function that counts how many times each of several search terms appears in a log file gets slower the more search terms are passed in — even though the file itself never changes between searches.

Root cause

Chapter 13 covered reading a file with open()/.read(). Disk I/O is slow relative to in-memory operations — re-opening and re-reading the entire file once per search term repeats that slow I/O work unnecessarily, when the file's contents haven't changed at all between calls.

Fix

Read the file into memory exactly once, then run every search against that in-memory string — I/O happens once regardless of how many search terms there are.

issue_18_reread_file.py 
import time

def count_matches_slow(path, needles):
    counts = {}
    for needle in needles:
        with open(path) as f:  # re-opens and re-reads the file every time
            content = f.read()
        counts[needle] = content.count(needle)
    return counts

def count_matches_fast(path, needles):
    with open(path) as f:
        content = f.read()  # read once
    counts = {}
    for needle in needles:
        counts[needle] = content.count(needle)
    return counts

# set up a sample log file so this script is runnable on its own
with open("access.log", "w") as f:
    for i in range(5000):
        f.write(f"line-{i}\n")

needles = [f"line-{i}" for i in range(0, 500, 50)]

start = time.perf_counter()
a = count_matches_slow("access.log", needles)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = count_matches_fast("access.log", needles)
fast_time = time.perf_counter() - start

print(a == b)
print(f"re-read file per needle: {slow_time:.4f}s")
print(f"read file once:          {fast_time:.4f}s")
actual output (Python 3.14.4, run against a 5,000-line log file) -- your own numbers will vary by machine and disk

#19

Materializing a full list when the loop could stop early

Symptom

A "find the first value over a threshold" function is dramatically slower than it should be on a large dataset, even though it usually only needs to look at the first handful of values before finding a match.

Root cause

Chapter 17 covered generators as lazy, on-demand value production. A list comprehension does the opposite: it computes and stores every value up front, before the loop even starts — so even though the loop below it exits early once a match is found, all the other work (doubling every remaining number that's never even inspected) already happened.

Fix

Use a generator expression instead of a list comprehension when the consumer might stop early. A generator produces values one at a time, on demand — once the loop finds its match and returns, no further values are ever computed.

issue_19_materialize_generator.py 
import time

def find_first_over_slow(numbers, threshold):
    doubled = [n * 2 for n in numbers]  # computes EVERY value up front
    for d in doubled:
        if d > threshold:
            return d
    return None

def find_first_over_fast(numbers, threshold):
    doubled = (n * 2 for n in numbers)  # lazy -- computes on demand
    for d in doubled:
        if d > threshold:
            return d
    return None

numbers = list(range(2000000))

start = time.perf_counter()
a = find_first_over_slow(numbers, 100)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = find_first_over_fast(numbers, 100)
fast_time = time.perf_counter() - start

print(a == b)
print(f"list comprehension (materializes all): {slow_time:.4f}s")
print(f"generator expression (lazy, stops early): {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
True
list comprehension (materializes all): 0.2775s
generator expression (lazy, stops early): 0.0000s

The single biggest speedup in this category, and it's a direct callback to Chapter 25's generators-vs-lists comparison — the same underlying idea (don't build what you don't need) showing up as a memory win there and a speed win here.

#20

Checking membership against dict.keys() instead of the dict itself

Symptom

A filtering function using if key in some_dict.keys() is a little slower than the same check written without .keys() — small, but consistent and free to fix.

Root cause

Chapter 8 covered dict membership (key in some_dict) as already O(1) average case — it doesn't need .keys() at all. Writing .keys() explicitly adds an unnecessary view-object layer that in then has to go through, for no benefit.

Fix

Drop the .keys() — key in some_dict checks the keys by default, is exactly as correct, and has one less layer of indirection.

issue_20_keys_membership.py 
import time

def filter_active_slow(user_ids, status_by_id):
    active = []
    for uid in user_ids:
        if uid in status_by_id.keys():  # unnecessary .keys()
            if status_by_id[uid] == "active":
                active.append(uid)
    return active

def filter_active_fast(user_ids, status_by_id):
    active = []
    for uid in user_ids:
        if uid in status_by_id:  # dict membership is already O(1)
            if status_by_id[uid] == "active":
                active.append(uid)
    return active

status_by_id = {i: ("active" if i % 2 == 0 else "inactive") for i in range(200000)}
user_ids = list(range(200000))

start = time.perf_counter()
a = filter_active_slow(user_ids, status_by_id)
slow_time = time.perf_counter() - start

start = time.perf_counter()
b = filter_active_fast(user_ids, status_by_id)
fast_time = time.perf_counter() - start

print(a == b)
print(f"'in dict.keys()': {slow_time:.4f}s")
print(f"'in dict':         {fast_time:.4f}s")
actual output (Python 3.14.4) -- your own numbers will vary by machine 
True
'in dict.keys()': 0.0231s
'in dict':         0.0161s
Quick recap
  • Nested loops and repeated in checks against a growing list/being scanned are the classic sign of an accidental O(n²) algorithm — a set or dict almost always fixes it.
  • Not every fix is a Big-O improvement — some (regex compilation, avoiding .keys(), hoisting len() out of a loop) just remove small, repeated overhead, and it's worth knowing the difference.
  • Generators (Chapter 17) aren't just a memory optimization (Chapter 25) — they're also a speed win whenever a consumer might stop early, since unconsumed values are never computed at all.
In the Real World (Category 2)
  • "Make it faster" without measuring first is how engineers waste time optimizing the wrong thing — every fix in this category was verified with an actual timer, not assumed.
  • The nested-loop-lookup pattern (issue #14) is one of the single most common real-world performance bugs — it shows up constantly in code that joins two datasets without first indexing one of them by key.
  • Knowing which fixes are dramatic (O(n²) → O(n)) vs. modest (removing small per-call overhead) helps prioritize: fix the algorithmic issues first, since that's where the real wins are.

Points to Remember (Categories 1-2)

  • KeyError, IndexError, TypeError, AttributeError, ZeroDivisionError, RecursionError, and JSONDecodeError each have a specific, predictable trigger — recognizing the pattern is faster than reading the whole traceback from scratch every time.
  • A function that implicitly returns None on some code path (Chapter 10) is one of the most common sources of a downstream AttributeError far from where the real problem started.
  • Bare except: and overly broad except Exception: are the two patterns most likely to turn a real bug into a silent, wrong-but-plausible result — always catch the most specific exception you actually expect.
  • A repeated in check against a list, or a nested loop joining two datasets, are the two most common sources of an accidental O(n²) algorithm — a set or dict almost always fixes both.
  • Caching (a dict keyed by whatever determines the result) is only correct when the same input reliably produces the same output — check that assumption before reaching for a cache.
