# ops — Capstone 3: Ops Toolkit

Three tools built separately across this course — the Chapter 13
Log-File Analyzer, the Chapter 18 Resume Keyword Scanner, and the
Chapter 19 Weather & Quote-of-the-Day Fetcher — unified into **one
installable, multi-subcommand CLI package** called `ops`. Instead of
three `python3 starter.py` scripts with three different menu loops and
three different error-handling habits, this ships one real command with
one consistent interface, one install step, and one `--help` system a
reader can discover the whole tool from.

## Subcommand tree

```text
ops
├── logs
│   └── analyze <file>              [--level] [--keyword] [--export FILE]
├── scan
│   ├── resume <file> --keywords a,b,c
│   └── contacts <file>              (email/phone extraction)
└── fetch
    ├── weather <city> [--simulate-error]
    └── quote [--index N]
```

Every subcommand also accepts a top-level `-v`/`--verbose` flag
(`ops -v logs analyze ...`) that raises logging from INFO to DEBUG
across *every* subcommand, not just one.

## Architecture: engine vs. dashboard

```text
ops/
├── pyproject.toml       <- [project.scripts] ops = "ops.cli:main"
├── requirements.txt     <- stdlib only: empty, with a comment (see below)
├── ops/
│   ├── __init__.py      <- package version + the shared OpsError exception
│   ├── cli.py            <- the ONLY file that touches argparse/print()/sys.exit
│   ├── logs.py           <- Ch13 engine (zero print()/input())
│   ├── scan.py            <- Ch18 engine (zero print()/input())
│   ├── fetch.py           <- Ch19 engine (zero print()/input())
│   └── formatting.py      <- shared output helpers (format_table/format_kv/format_list)
├── sample_data/           <- server.log + resume.txt used in the examples above
└── tests/                 <- stdlib unittest suite (18 tests, `python -m unittest discover -s tests`)
```

`ops/logs.py`, `ops/scan.py`, and `ops/fetch.py` never `import argparse`
and never call `print()`/`input()` — they're plain functions (and one
class, `ResumeScanner`) that take arguments and return values, or raise
an exception. That means all three engines could be imported and reused
by a future script, a test suite, or even a web API with zero changes —
the same "engine talks to data, dashboard talks to the terminal"
separation Chapter 31's `taskbox` established.

## One shared convention, not three stapled-together tools

- **One error-handling convention.** Every domain module raises either
  a plain `FileNotFoundError` (missing file — the exception `open()`
  already raises) or a shared `OpsError` (defined once, in
  `ops/__init__.py`, imported by all three engine modules) for anything
  else: an empty search keyword, an empty resume-keyword list, a
  simulated bad weather response, an out-of-range quote index, or a
  write failure during CSV export. `ops/cli.py` has exactly one
  `try/except (OpsError, FileNotFoundError)` per handler — never three
  different ad hoc error-printing styles.
- **One output convention.** `ops/formatting.py` provides
  `format_table`, `format_kv`, and `format_list` — used by `logs`,
  `scan`, *and* `fetch` output alike, so no domain invents its own
  alignment or spacing rules. (Caught in review: an earlier draft had
  `fetch.py` hand-rolling its own weather/quote strings instead of
  routing through these helpers — exactly the "stapled together"
  failure mode this section warns about. Fixed by rewriting
  `format_weather()`/`format_quote()` to call `format_kv`/`format_list`
  internally.)
- **One `-v`/`--verbose` flag**, parsed once at the top level, that
  raises the `"ops"` logger from INFO to DEBUG for every subcommand
  (see the verbose section above).
- **One `--help` system**, generated entirely by `argparse`'s
  subparsers-of-subparsers (`ops` → `logs`/`scan`/`fetch` → each
  domain's actions), with no hand-written help text duplicated anywhere.

## Why `requirements.txt` is empty

`ops` is stdlib-only — `argparse`, `logging`, `re`, `csv`, `json`,
`pathlib` are all that's needed. `requirements.txt` is kept (not
omitted) as a documented, deliberate statement that this tool has zero
third-party runtime dependencies, following `taskbox`'s own precedent
from Chapter 31.
